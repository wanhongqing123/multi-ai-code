# Diff 审查界面中的 Markdown 预览设计

## 背景

当前 Diff 审查界面统一以源码 diff 的方式展示文件内容。对于 `.md` / `.markdown` 文件，这种展示方式能看出文本改动，但不能直接看出渲染后的阅读效果。仓库内已经存在可复用的 Markdown 渲染能力：

- `react-markdown`
- `remark-gfm`
- 统一的 `.md-rendered` 样式

因此本次改动的目标不是替换现有 diff，而是在保留逐行审查能力的前提下，为 Markdown 文件补充一个渲染预览视图。

## 目标

- 对 `.md` / `.markdown` 文件，在 Diff 审查界面中提供 `Diff` / `Markdown Preview` 两种视图。
- 默认仍进入 `Diff` 视图，不改变当前审查工作流。
- `Markdown Preview` 视图展示旧版本与新版本的渲染结果，帮助用户检查结构、排版、列表、表格、代码块等 Markdown 呈现效果。

## 非目标

- 不替换现有源码 diff 视图。
- 不为非 Markdown 文件提供预览模式。
- 不在 Markdown 预览中支持文本选择批注、行号、搜索命中跳转或 hunk 跳转。
- 不引入新的 Markdown 渲染库或主题体系。

## 方案概览

在 `src/components/DiffViewerDialog.tsx` 内为当前选中文件增加一个“文件视图模式”状态：

- 非 Markdown 文件：只显示现有 `Diff` 视图。
- Markdown 文件：显示 `Diff` / `Markdown Preview` 切换。

当用户切换到 `Markdown Preview` 时，主内容区不再渲染逐行 diff，而是渲染一个左右对照的 Markdown 预览：

- 左侧：旧版本文档内容
- 右侧：新版本文档内容

这样既保留“看改动”的能力，也补上“看渲染结果”的能力。

## 交互设计

### 1. 视图切换

- 切换控件仅在当前文件扩展名为 `.md` 或 `.markdown` 时显示。
- 选项为：
  - `Diff`
  - `Markdown Preview`
- 默认选中 `Diff`。
- 当用户从 Markdown 文件切换到非 Markdown 文件时，界面自动退回 `Diff` 视图，避免无效状态残留。

### 2. Diff 视图

- 保持当前行为不变。
- 继续支持：
  - 文件树选择
  - hunk 跳转
  - 搜索
  - 文本选择后添加批注
  - 右侧批注列表与定位

### 3. Markdown Preview 视图

- 使用现有 `react-markdown + remark-gfm + .md-rendered` 渲染。
- 采用双栏对照布局：
  - 左栏标题：`Old`
  - 右栏标题：`New`
- 特殊情况：
  - 新增文件：左栏显示空态，右栏显示渲染结果
  - 删除文件：左栏显示渲染结果，右栏显示空态
  - 无可渲染内容：显示明确空态文案
- 预览区域为只读，不支持批注浮层和文本选区写回。

## 数据设计

当前 `DiffViewerDialog` 已把 unified diff 解析为 `DiffFile.lines`。实现 Markdown 预览时，直接从该结构推导旧文本与新文本：

- 旧文本由 `context` + `del` 行组成
- 新文本由 `context` + `add` 行组成
- `hunk` 与 `meta` 行不进入文档正文

这意味着不需要额外请求原始文件内容，也不需要新增后端 IPC。

建议新增一个纯函数辅助逻辑，职责是从 `DiffFile` 构造：

- `oldText`
- `newText`

该函数应保持纯净、可测试，并且不依赖 React 状态。

## 组件边界

建议拆出一个小型只读组件，例如 `MarkdownDiffPreview`，由 `DiffViewerDialog` 调用：

- 输入：
  - `filePath`
  - `oldText`
  - `newText`
- 输出：
  - 左右双栏 Markdown 预览 DOM

这样做的原因：

- 避免把 `DiffViewerDialog` 的渲染分支继续堆大
- 便于单独测试 Markdown 预览结构
- 便于后续为预览补充局部样式，而不污染现有 diff 行渲染逻辑

## 样式设计

复用 `.md-rendered` 的基础排版，并在 DiffViewer 区域补一组专用样式：

- 预览容器
- 左右栏布局
- 栏头
- 空态块
- 预览滚动区域

布局要求：

- 桌面端左右分栏
- 窄屏下可退化为上下堆叠，保持可读性
- 不沿用代码 diff 的等宽字体；Markdown 正文应使用普通正文排版

## 错误处理

- 如果当前文件不是 Markdown，不显示预览切换入口。
- 如果 diff 解析结果为空或当前文件缺失，继续沿用现有空态，不新增特殊异常分支。
- 如果 Markdown 含有不安全链接或图片，沿用现有组件中的安全策略：
  - 链接仅允许 `https:`, `http:`, `mailto:`, `#`
  - 图片仅允许 `https:` 或 `data:image/`

## 测试设计

按 TDD 实现时，至少覆盖以下测试：

- Markdown 文件时显示视图切换；非 Markdown 文件时不显示。
- 进入 `Markdown Preview` 后，渲染左右两栏预览。
- 从 `DiffFile` 推导 `oldText` / `newText` 的纯函数能正确处理：
  - 普通修改
  - 新增文件
  - 删除文件
- 预览视图下不渲染逐行 diff 主体。
- 预览视图能正确渲染标题、列表、表格或代码块中的至少一种 GFM 内容。

## 实现步骤

1. 为 Markdown 预览文本提取逻辑补纯函数测试。
2. 实现文本提取纯函数，并确认测试先失败后通过。
3. 为 `DiffViewerDialog` 的 Markdown 文件视图切换补组件测试。
4. 实现 `Diff` / `Markdown Preview` 切换逻辑。
5. 抽出并接入 `MarkdownDiffPreview` 组件。
6. 补充 DiffViewer 专用样式。
7. 运行相关测试，确认现有 diff 功能未回归。

## 风险与取舍

### 1. 从 diff 重建文档文本的准确性

该方案依赖 unified diff 中的 `context` / `add` / `del` 行重建文档正文。对标准 git diff 来说这是可行的，但必须避免把 hunk header 或 meta 行混入正文，否则预览会被污染。

### 2. 搜索与批注行为差异

Markdown 预览是只读视图，而现有 diff 是可批注视图。这个差异是有意设计，不应在首版里强行统一，否则复杂度会明显上升。

### 3. 布局重量

预览是双栏渲染，长文档时会比普通文本更重。首版优先复用现有渲染库和样式，不做虚拟化；如果后续出现性能问题，再单独优化。

## 最终结论

采用“保留源码 diff，并为 Markdown 文件补一个可切换的双栏预览视图”的方案。它对现有行为破坏最小，能直接解决“`.md` 文件在 Diff 审查中缺少渲染视角”的问题，并且能充分复用仓库里现成的 Markdown 渲染能力与样式。
