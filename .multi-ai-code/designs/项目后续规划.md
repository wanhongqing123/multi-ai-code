# Diff 审查外部 AI 建议导入与 AI Markdown 文件保护设计

## 背景

当前 Diff 审查界面已经支持：

- 查看逐文件 diff
- 添加人工批注
- 把批注结果回灌给 AICLI

但还缺两项能力：

1. 无法把外部 AI 生成的 review 建议导入到 Diff 审查流中，导致用户需要手工搬运建议。
2. Claude 会话当前会把系统注入内容写到目标仓库根目录的 `CLAUDE.md`，这会污染项目仓库，也可能覆盖或干扰仓库原本已经存在的 AI 协作文档。

本次改动的目标，是在不破坏现有 Diff 审查主流程的前提下，把外部 AI review 建议纳入统一审查入口，并把平台自己的 AI 注入文件与仓库业务文件彻底隔离。

## 目标

- 在 Diff 审查界面支持导入外部 AI review 文本文件。
- 把导入内容解析为逐条建议列表，而不是整段自由文本。
- 支持逐条或批量把建议发送给 AICLI 判断。
- 对每条建议给出明确结论和理由：
  - `采纳`
  - `不采纳`
  - `需要人工确认`
- AICLI 的判断结果只写回界面状态，不直接改代码。
- 不再修改仓库中的 `CLAUDE.md`、`AGENT.md`、`AGENTS.md`，并尽量不修改项目仓库中其他 AI 相关 Markdown 文件。

## 非目标

- 不把外部 AI 建议自动转成正式批注。
- 不自动根据 AI 建议修改代码或生成补丁。
- 不尝试支持所有 review 文本格式的高精度结构化解析。
- 不把外部 AI 建议长期持久化成项目数据模型的一部分；首版以当前审查会话内可用为主。
- 不改动仓库里用户已有的 AI 协作文档内容，即使文件不存在也不新增同名仓库根目录文件。

## 方案概览

本次拆成两个独立但相关的子系统：

### 1. Diff 审查中的外部 AI 建议侧栏

在 `DiffViewerDialog` 中新增一个“外部 AI 建议”区域，用于：

- 导入外部 review 文件
- 展示解析后的建议列表
- 触发单条或批量的 AICLI 判断
- 展示每条建议的状态与理由

### 2. AI 注入文件隔离策略

调整 Claude 会话的系统注入方式，不再向目标仓库根目录写入 `CLAUDE.md`。平台生成的 AI 提示文件统一放到平台自有目录，例如：

- 会话工作目录下的 `.injections/`
- 或 `.multi-ai-code/` 的受控子目录

CLI 需要读取这些内容时，通过命令参数、stdin、会话级临时文件路径或平台侧拼接提示完成，而不是通过修改仓库文件达成自动加载。

## 设计一：外部 AI review 导入与逐条判断

### 1. 导入入口

在 Diff 审查界面中新增“导入外部 AI review”操作入口。

首版支持导入：

- `.md`
- `.markdown`
- `.txt`

导入方式继续复用现有 Electron 文件选择能力，不新增浏览器端直接读本地文件的特殊路径。

### 2. 建议列表模型

导入后的文本会被解析成建议项列表。每条建议至少包含：

- `id`
- `sourceLabel`
  - 导入文件名或用户可识别来源
- `rawText`
  - 该条建议的原文
- `pathHint`
  - 可选，解析出的文件路径线索
- `lineHint`
  - 可选，解析出的行号线索
- `status`
  - `idle`
  - `accepted`
  - `rejected`
  - `needs-human`
  - `error`
- `decisionReason`
  - AICLI 返回的理由
- `linkedDiffFile`
  - 可选，和当前 diff 文件树匹配后的结果

这里的 `accepted / rejected / needs-human` 是 AICLI 给出的判断结论，不代表代码已经改完，也不代表建议已经关闭。

### 3. 文本拆分策略

首版不追求复杂 NLP 解析，采用可预测的规则拆分：

- 优先按 Markdown 列表项拆分：
  - `- `
  - `* `
  - `1. `
- 如果存在明显的小节标题，则标题作为分组上下文，但最终展示仍然是逐条建议。
- 如果整份文本没有可识别列表，则按空行分段。
- 过滤纯标题、纯分隔线、空白段落。

解析原则是宁可保守拆分，也不要把多条建议错误合并成一条超长文本。

### 4. 文件与行号线索提取

从每条建议中尽量提取：

- 类似 `src/foo.ts`、`electron/main.ts` 的路径片段
- 类似 `L42`、`line 42`、`42-48` 的行号线索

提取成功后：

- 如果能匹配当前 diff 中的文件路径，则写入 `linkedDiffFile`
- 如果不能匹配，保留为文本线索，不阻塞导入

首版只做弱联动，不做强依赖。也就是说，即使没有路径和行号，也必须允许建议存在并发送给 AICLI 判断。

### 5. 界面交互

建议区域至少包含以下能力：

- 展示导入来源和建议总数
- 逐条展开查看建议原文
- 展示当前状态标签
- 展示 AICLI 判断理由
- 单条“发送给 AICLI 判断”
- 批量“全部发送给 AICLI 判断”

如果建议已经有判断结果，允许用户重新发送，以便在 diff、上下文或提示变化后重新判断。

### 6. AICLI 判断协议

发送给 AICLI 的内容需要结构化，至少包含：

- 当前审查目标的上下文
- 建议原文
- 可选的文件路径和行号线索
- 如果已经匹配到 diff 文件，则附带相关文件路径
- 明确要求 AICLI 只返回：
  - 结论
  - 理由

返回格式应收敛成稳定结构，避免让前端从自由文本里二次猜测。例如：

- `decision`
- `reason`

其中 `decision` 只允许：

- `accepted`
- `rejected`
- `needs-human`

如果 AICLI 返回无法解析的内容，前端把该条状态置为 `error`，并展示原始失败说明，而不是悄悄吞掉。

### 7. 与现有批注流的关系

外部 AI 建议和现有人工批注是并列能力，不互相替代：

- 人工批注仍然来自用户在 diff 中选中文本后创建的注释。
- 外部 AI 建议来自导入文件，初始不自动落到行级批注系统。
- AICLI 对外部建议的判断结果，只作为审查参考，不自动生成 patch、不自动生成 comment thread。

这样做的原因是，外部建议往往缺少足够精确的行级定位，强行灌进批注模型会降低数据质量。

### 8. 错误处理

- 文件导入失败：给出明确错误提示，不保留半初始化状态。
- 文本解析后无有效建议：展示“未识别到可导入建议”的空状态。
- 单条判断失败：只影响该条建议状态，不阻塞其他建议。
- 批量判断时部分失败：逐条保留成功与失败结果，不做全量回滚。

## 设计二：AI 相关 Markdown 文件保护策略

### 1. 保护范围

首要保护对象：

- `CLAUDE.md`
- `AGENT.md`
- `AGENTS.md`

同时遵循更强的总原则：

- 平台自身不应主动修改目标仓库中的 AI 协作文档
- 如果某些仓库已经存在其他 AI 相关 Markdown 文件，也尽量不碰它们

### 2. 当前问题

当前 Claude 会话通过在目标仓库工作目录写入 `CLAUDE.md` 来实现自动加载系统提示。这种做法有三个问题：

- 会污染项目仓库工作树
- 可能覆盖仓库中原有的 `CLAUDE.md`
- 会让平台自己的系统约束和仓库作者自己的协作规范混在一起

### 3. 新策略

Claude 会话改为与 Codex 注入策略保持一致：平台生成的系统提示文件写入平台自有路径，不写入仓库根目录。

建议路径：

- `<session-workdir>/.injections/`

如有必要，也可放在：

- `.multi-ai-code/runtime-injections/`

但必须满足两个条件：

1. 文件属于平台运行时产物，而不是仓库业务文件。
2. CLI 读取方式不依赖仓库根目录自动发现 `CLAUDE.md`。

### 4. 兼容行为

如果某个 CLI 的确只支持从约定文件名自动发现提示，则首选方案不是“继续往仓库里写文件”，而是：

- 调整启动命令，让 CLI 指向平台生成的注入文件
- 或把注入文本通过 stdin / 参数传入
- 或在平台层把系统提示拼接进发送给 CLI 的上游消息

只有在完全没有替代机制时，才考虑降级方案。但即使降级，也必须遵守：

- 如果仓库中已有 `CLAUDE.md`、`AGENT.md`、`AGENTS.md`，绝不覆盖
- 默认也不新增这些文件

也就是说，首版实现应以“完全停止修改仓库内 AI Markdown 文件”为目标，而不是“存在时不覆盖，不存在时新建”。

### 5. 清理与生命周期

平台生成的注入文件应当：

- 放在可控目录
- 随会话创建
- 会话结束或目录清理时可删除

这样可以避免在项目仓库中残留运行时噪声文件。

## 数据与组件边界

### 前端

建议新增一个独立的小组件，例如：

- `ExternalAiReviewPanel`

职责：

- 文件导入入口
- 建议列表展示
- 状态展示
- 单条/批量触发判断

`DiffViewerDialog` 负责：

- 挂载该面板
- 提供当前 diff 上下文
- 维持和现有审查界面的布局一致性

### 解析逻辑

建议把“文本拆分与线索提取”做成独立纯函数，便于测试。例如：

- `parseExternalReviewSuggestions(text: string, sourceLabel: string)`

输出结构化建议列表。

### 后端 / 会话层

建议把“AICLI 判断外部建议”的调用做成独立接口，避免直接复用现有“提交人工批注”协议时混入太多分支判断。这样后续如果要支持别的 AI 提供者，也更容易扩展。

## 测试设计

至少覆盖以下测试：

### 1. 解析测试

- Markdown 列表可被拆成多条建议
- 空行分段文本可被拆成多条建议
- 可正确提取文件路径线索
- 可正确提取行号线索
- 无有效建议时返回空列表

### 2. 前端交互测试

- 能导入并展示建议列表
- 单条发送判断后可展示结论与理由
- 批量发送时每条状态独立更新
- 判断失败时只标记对应条目为错误
- 无路径线索的建议仍可正常判断

### 3. 注入保护测试

- Claude 会话不再向仓库根目录写 `CLAUDE.md`
- 仓库中已有 `CLAUDE.md` 时不会被修改
- 仓库中已有 `AGENT.md` / `AGENTS.md` 时不会被修改
- 新注入文件写入平台自有目录

## 实施顺序

1. 先补外部 review 解析函数测试。
2. 实现解析逻辑并验证拆分、路径提取、行号提取。
3. 为 Diff 审查面板补外部建议 UI 测试。
4. 接入导入、列表状态与 AICLI 判断流程。
5. 为 Claude 注入路径补保护性测试。
6. 修改会话注入实现，停止写仓库根目录 `CLAUDE.md`。
7. 跑相关测试与全量验证。

## 风险与取舍

### 1. 外部 review 文本格式不统一

不同 AI 工具导出的 review 格式差异很大，因此首版只能做规则化拆分，不能承诺“任意格式都高精度解析”。这里的取舍是优先保证可用和可解释，而不是追求过度智能。

### 2. 判断结果不是最终事实

AICLI 给出的“采纳 / 不采纳 / 需要人工确认”本质上仍然是审查建议，不应被误认为最终决策。因此界面文案需要明确这是 AI judgement，而不是自动执行结果。

### 3. Claude 自动加载兼容性

如果现有 Claude CLI 代码路径强依赖仓库根目录 `CLAUDE.md`，改成外部注入后可能需要同时调整启动参数或消息拼接逻辑。这是必要改动，因为用户要求已经明确禁止再动仓库内这些文件。

## 最终结论

采用“在 Diff 审查界面新增外部 AI 建议侧栏，并把建议解析成逐条项目交给 AICLI 判断”的方案。同时，把 Claude 的系统注入从仓库根目录 `CLAUDE.md` 迁移到平台自有运行时目录，明确不再修改仓库中的 `CLAUDE.md`、`AGENT.md`、`AGENTS.md` 等 AI 协作文档。
