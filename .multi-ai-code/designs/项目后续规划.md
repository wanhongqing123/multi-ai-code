# 习惯学习智能体（Skill 自动建议）设计

## 背景

当前 Multi-AI Code 已有两个产品级 AI 角色：

- 主会话 AI：驱动主界面 PTY 终端
- 仓库查看分析 AI：在 `仓库查看` 窗口里做代码标注分析

用户在长期使用过程中，会重复输入大量相似的 prompt，重复触发相似的工作流。这些重复模式当前需要用户自己手动整理到顶栏的"模板"功能里，门槛高、容易遗忘。

本次新增**第三个产品级 AI 角色：习惯学习智能体（Habit Agent）**，定位是被动观察 + 定期汇总 + 主动建议：

- 被动观察：在用户操作过程中静默采集 App 内行为信号
- 定期汇总：按周期对采集到的行为做本地聚合分析
- 主动建议：把聚合结果交给 AI CLI，生成"候选 prompt 模板"，由用户决定是否采纳进现有"模板"系统

## 目标

- 新增"习惯学习智能体"作为第三个独立 AI 角色，与主会话 AI、仓库查看分析 AI 并列。
- 仅在本 App 内部采集行为信号，不监控本机操作系统其他活动。
- 行为数据默认只存本地，仅在 skill 生成时把**最小聚合摘要**发给 CLI。
- 定期（每天/每周）汇总，生成"候选 skill"列表，由用户决定采纳/编辑/丢弃。
- 采纳后的 skill 直接进入现有顶栏"模板"系统，复用现有调用入口。
- 提供清晰的开关、采集源管理、保留期管理、清空数据入口。

## 非目标

- 不监控系统级行为（不抓键盘、鼠标、窗口活动、其他进程）。
- 不读取 shell 历史、git 提交、本机其他目录的文件变动。
- 不实时弹窗打断用户。
- 不自动把生成的 skill 直接执行或注入会话；所有 skill 必须用户显式采纳后才进入模板列表。
- 不为该智能体设计独立的对话窗口；它的产物只有"候选 skill 列表"。
- 不做云同步、不做跨设备数据迁移（行为数据始终本地）。
- v1 不为本地数据库引入 SQLCipher 等应用层加密；只依赖操作系统盘加密。如果用户需要更强保护，需在后续版本单独立项。
- 不重写现有"模板"系统的数据结构；保持 `{id, name, body}` 三字段兼容。

## 方案概览

本方案分四个子系统：

1. **信号采集层（Collector）**：在 App 内现有关键交互点静默记录事件到本地 SQLite。
2. **本地聚合层（Aggregator）**：周期性对原始事件做无 LLM 的聚类与打分。
3. **生成层（Generator）**：把聚类摘要交给 CLI，生成候选 prompt 模板。
4. **审阅 UI**：候选 skill 列表 + 采集设置 + 数据查看与清空入口。

### 数据流

```
用户操作 → Collector 钩子 → 本地 SQLite (habit_events 表)
                                  │
                                  ▼
            周期调度（每天/手动触发）
                                  │
                                  ▼
       Aggregator（纯本地）：聚类 + 打分 → top-K 候选簇
                                  │
                                  ▼
       Generator：构造最小摘要 → 交给习惯学习 AI CLI
                                  │
                                  ▼
        CLI 返回 {title, body, meta} → 写入 skill_candidates 表
                                  │
                                  ▼
            审阅 UI（用户决策：采纳 / 编辑 / 丢弃 / 暂不处理）
                                  │
                                  ▼
                采纳 → 写入现有 localStorage 模板系统
```

## 设计一：信号采集层

### 1. 采集范围（仅 App 内）

| 事件类型 | 来源代码 | 记录字段（payload） |
|---|---|---|
| `pty_cmd` | `electron/cc/PtyCCProcess.ts` | 用户在主会话 PTY 中提交的命令行 |
| `ai_prompt_main` | 主会话提交点 | 发送给主会话 AI 的 prompt 原文 |
| `ai_prompt_repo` | `electron/repo-view/repoAnalysisManager.ts` | 仓库查看窗口提交的分析 prompt |
| `diff_annotation` | `src/components/DiffViewerDialog.tsx` | Diff 批注文本 |
| `repo_view_annotation` | `src/repo-view/AnalysisPanel.tsx` | 仓库查看里的代码标注文本 |
| `template_used` | `src/components/TemplatesDialog.tsx` 注入入口 | 被调用的模板 id 与 name |
| `plan_imported` | `electron/orchestrator/plans.ts` | 外部方案导入的源类型（仅类型，不含路径） |

不采集：
- 终端输出内容
- AI 返回的文本
- 文件内容
- 完整的方案文件正文

理由：采集 input 侧已经能描述"用户的意图"，output 侧主要是噪声，且增大数据体积与隐私风险。

### 2. 存储

新增 SQLite 表（写入现有 `electron/store/db.ts` 维护的 db 文件）：

```sql
CREATE TABLE IF NOT EXISTS habit_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts INTEGER NOT NULL,
  kind TEXT NOT NULL,
  payload TEXT NOT NULL,
  project_id TEXT,
  repo_path TEXT,
  source_window TEXT
);
CREATE INDEX IF NOT EXISTS idx_habit_events_ts ON habit_events(ts);
CREATE INDEX IF NOT EXISTS idx_habit_events_kind ON habit_events(kind);
```

- `payload` 为 JSON 字符串
- `project_id` / `repo_path` 用于后续按项目维度分组（v1 不强分桶但记录该维度）
- `source_window` 取值如 `main` / `repo-view` / `diff-review`

### 3. 默认开关与保留期

- 主开关默认 **开启**。首次开启会一次性弹出告知面板，说明采集范围、不采集什么、数据去向、如何关闭与清空。告知面板有"我知道了"和"立即关闭采集"两个出口。
- 每个事件类型有独立子开关，默认全开。
- 默认保留期 90 天，超期自动按 ts 删除。
- 用户可在设置面板里：
  - 切主开关
  - 切每个事件类型的子开关
  - 调整保留期（30 / 60 / 90 / 180 天）
  - 查看最近 N 条原始事件
  - 一键清空所有 habit_events 记录

### 4. 采集钩子的实现方式

新增 `electron/habit/collector.ts`，对外暴露：

```ts
recordEvent(kind: HabitEventKind, payload: object, ctx?: { projectId?: string; repoPath?: string; sourceWindow?: string }): void
```

在上述各采集点处调用。调用方不需要关心：

- 主开关是否开启（关闭时函数直接 no-op）
- 子开关是否开启
- DB 是否就绪

把所有判断收敛在 `recordEvent` 内部，最大化降低改动面，避免污染现有业务代码。

## 设计二：本地聚合层

### 1. 调度

新增 `electron/habit/scheduler.ts`：

- App 启动后等待 N 秒（默认 60s）跑一次轻量聚合，把"距上次完整聚合超过 24h"的状态置 dirty
- 每隔 24h 跑一次完整聚合
- 用户可在 UI 手动触发"立即分析"

聚合本身不调 LLM，是纯本地计算，CPU 与磁盘开销可控。

### 2. 聚合算法 v1

新增 `electron/habit/aggregator.ts`，纯函数，便于测试。

输入：最近 30 天的 `habit_events` 列表  
输出：top K 个候选簇，每个含 `{ representativeText, sampleEventIds, size, score, kindDistribution }`

算法：

1. 按 `kind` 分组，避免跨类型聚类（命令行和 prompt 文本天然不同）。
2. 对每组内的文本，做轻量预处理：
   - 去掉路径、行号、临时数字
   - 中英文混合分词（按空白 + 标点切，不强求 NLP 库）
3. 计算 shingles（k=3 词组），用 Jaccard 相似度 ≥ 0.4 做并查集聚类。
4. 簇得分 = `cluster_size * recency_decay`，其中 `recency_decay = exp(-Δdays / 14)`。
5. 过滤 `size < 3` 的簇。
6. 按得分排序，取 top K（默认 10）。

特点：

- 完全本地，无外部依赖
- 算法朴素但可解释、可测试
- 后续可在不破坏接口的前提下替换为 MinHash / TF-IDF / 嵌入向量

### 3. 跨项目维度

v1 不按 `project_id` 强分桶，所有项目的事件进同一个池。但簇结构里附带 `projectDistribution`，让生成层能在提示词里告诉 CLI：

- 这是"跨多项目通用模式"还是"专属某项目的模式"
- 不同模式对应不同的 skill 命名风格（前者更通用，后者更具体）

## 设计三：生成层

### 1. 与现有 AI 配置的关系

**直接复用主会话 AI 配置**，不引入第三套配置。

- 习惯学习智能体每次生成 skill 时，调用的 CLI 与主会话相同
- 用 binary / args / env 也完全沿用主会话设置
- `AI 设置` 对话框保持现状（仍为主会话 + 仓库查看 两套）
- 在习惯学习 UI 内部用一行小字注明：当前使用的 CLI 来自主会话 AI 设置

后续如有需要再拆出独立配置（例如允许用更便宜的模型跑批量分析），届时新增一个"使用独立配置"的开关即可，不破坏当前结构。

### 2. 发送给 CLI 的内容（B 模式：最小化导出）

新增 `electron/habit/generator.ts`。每个候选簇构造**单独一次**对话调用，发送内容严格限定为：

```
[簇描述]
- 类型: ai_prompt_main / pty_cmd / ...
- 出现次数: 12
- 时间跨度: 最近 18 天
- 涉及项目数: 3
- 是否跨项目通用: 是

[代表样本 3-5 条]
1. "帮我看看 src/foo.ts 的实现"
2. "解释 electron/main.ts 的会话管理逻辑"
3. ...

[要求]
基于以上重复模式，生成一个可复用的 prompt 模板。返回严格 JSON：
{ "title": "...", "body": "...", "variables": ["..."], "category": "...", "rationale": "..." }
```

注意：

- 不发完整事件列表，只发 3-5 条代表样本（取簇中心距离最小的几条）
- 不发原始时间戳、不发 project_id 原始值（用"涉及项目数"代替）
- 不发文件内容、不发 AI 返回内容

每次生成失败时记录失败原因，状态置为 `error`，不阻塞其他候选。

### 3. 候选 skill 存储

```sql
CREATE TABLE IF NOT EXISTS skill_candidates (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at INTEGER NOT NULL,
  cluster_kind TEXT NOT NULL,
  cluster_size INTEGER NOT NULL,
  source_event_ids TEXT NOT NULL,
  representative_samples TEXT NOT NULL,
  generated_title TEXT,
  generated_body TEXT,
  generated_meta TEXT,
  status TEXT NOT NULL DEFAULT 'pending',
  reviewed_at INTEGER,
  error_message TEXT
);
```

`status` 取值：`pending` / `accepted` / `edited` / `discarded` / `snoozed` / `error`

## 设计四：审阅 UI

### 1. 顶栏入口

新增按钮：**`Skill 学习`**，位置在顶栏现有 `模板` 旁边。

点击打开 `SkillStudioDialog`，包含两个 tab：

- **候选**：未审阅的 `pending` / `snoozed` 列表
- **采集**：开关、子开关、保留期、最近事件查看、清空

当前所用的 CLI（来自主会话 AI 设置）在对话框顶部以一行小字注明，附跳转到 `AI 设置` 的链接。

### 2. 候选 tab

每条候选展示：

- 生成时间
- 簇大小 + 类型
- AI 生成的 title / body / category / rationale
- 操作按钮：
  - `采纳` → 写入现有 localStorage 模板，status = `accepted`，关掉条目
  - `编辑后采纳` → 打开编辑器修改 title/body，提交后状态 `accepted`
  - `丢弃` → 状态置 `discarded`，从列表移除
  - `暂不处理` → 状态置 `snoozed`，N 天后重新出现
  - `查看证据` → 展开看本簇的代表样本与 source_event_ids 链接

- 在发送给 CLI 之前还有一个**预览发送内容**入口：让用户在生成前能看到本次将发给 CLI 的精确 payload（B 模式承诺的可视化兜底）。

### 3. 采纳后的模板写入

复用现有结构：

```ts
{
  id: `t_skill_${candidateId}`,
  name: generated_title,
  body: generated_body
}
```

由于现有模板存 localStorage、不在主进程，采纳逻辑放在渲染进程：渲染进程读到候选数据后调用现有 `load() / save()` 模式追加，不通过 IPC 复杂回传。

### 4. 采集 tab

包含：

- 主开关（默认关闭）
- 各 kind 子开关（默认全开）
- 保留期下拉
- "最近 100 条事件"只读列表（按 kind 着色，时间倒序）
- "清空所有数据" 按钮（二次确认）

### 5. 首启告知

App 升级到本版本后**首次进入主界面**时弹一次告知面板（因为采集默认开启，需要让用户知情）：

- 我们采集什么（7 个事件类型）
- 不采集什么（无系统级监控、不记 output 侧）
- 数据存哪里（本地 SQLite）
- 什么时候发出去、发什么（生成 skill 时只发最小聚合摘要）
- 可以随时关闭与清空

面板有两个按钮：
- `我知道了` → 维持默认开启状态，写入"已告知"标记
- `先关闭采集` → 关掉主开关，仍写入"已告知"标记，但不再每次启动都弹

告知只弹一次，状态写本地。

## 测试设计

### 1. Collector 测试

- 主开关关闭时 recordEvent 不写入 DB
- 子开关关闭时对应 kind 不写入
- 各种 payload 类型均能正确序列化
- 超出保留期的记录会被定期清理

### 2. Aggregator 测试

- 完全相同的文本会聚成一个簇
- 路径不同但模式相同的 prompt 会聚成一个簇（验证预处理去掉路径的能力）
- 簇大小 < 3 会被过滤
- recency_decay 对得分排序的影响符合预期
- 完全没有事件时返回空列表

### 3. Generator 测试

- 构造的 CLI payload 中**不含**原始 project_id
- payload 中样本数不超过 5 条
- CLI 返回非法 JSON 时候选 status 置 `error`
- CLI 调用失败时记录 error_message，不影响其他候选

### 4. UI 测试

- 候选列表正确展示与排序
- 采纳后的候选从列表移除并写入模板
- 编辑后采纳保留用户修改
- 暂不处理后会在指定天数后重新出现
- 预览发送内容显示与实际生成调用一致

### 5. 隐私不变量测试

- payload 序列化中不出现 `repo_path`、文件内容、AI 返回文本（即使作为子串）
- 主开关关闭后所有采集点都为 no-op
- 清空数据后 DB 表为空

## 实施顺序

1. DB schema + 迁移
2. Collector 接口与主开关/子开关逻辑（无 UI）
3. 在最少干扰的两个采集点（PTY 命令 + Diff 批注）先接入，验证可行
4. Aggregator 纯函数 + 完整测试
5. 采集 tab UI（先让用户能开关、看数据、清空）
6. Scheduler 与"立即分析"按钮
7. Generator + CLI 调用 + 失败处理
8. 候选 tab UI 与采纳流
9. 把剩余采集点接入（AI prompt、仓库查看标注、模板使用、方案导入）
10. 首启引导
11. 集成测试与隐私不变量测试
12. README 补充新角色说明

每完成一步在终端简述变更，按 CLAUDE.md 要求执行。

## 风险与取舍

### 1. 高频 ≠ 值得抽成 skill

朴素频次模型会把"输入回车"这种噪声选成 top 候选。缓解：

- 文本最小长度阈值（默认 ≥ 8 字符）
- 同一文本在 24h 内最多计 1 次
- 排除单词级别短文本（仅 pty_cmd 类）

### 2. 多项目共享 skill 容易产生不通用模板

跨多项目聚类时，如果 3 个项目实际上是 3 套语义不同的代码（前端/后端/桌面），生成的"通用"模板可能两边都不贴合。缓解：

- 在 generator 提示里明确告知 CLI"跨项目数 N、是否同类型"
- 让 CLI 在不确定时主动给"建议拆成两个变体"的提示
- 用户可在编辑后采纳里手工拆分

### 3. 隐私感知

即使是 B 模式，也要让用户在生成前能清楚看到"本次将发什么"。已通过预览发送内容入口缓解。

### 4. CLI 失败长尾

生成调用是异步、批量、可能数十次的过程。失败要：

- 单条失败不阻塞其他候选
- 错误信息保留在 skill_candidates.error_message
- UI 上有"重试本条"入口

### 5. 与现有 localStorage 模板系统耦合

采纳路径直接写 localStorage，绕过主进程 SQLite。这会带来：

- 跨设备同步缺失（与现有模板一致，不算回退）
- 多窗口同时打开时可能有竞争

v1 接受这个一致性级别。后续若把现有模板迁到 SQLite，再统一改造。

## 最终结论

新增"习惯学习智能体"作为第三个产品级 AI 角色。它被动观察 App 内行为、本地周期聚类、按 B 模式（最小化导出）调用 CLI 生成候选 prompt 模板，由用户审阅采纳后流入现有"模板"系统。所有数据默认本地、默认关闭采集、提供完整的开关与清空入口，并在生成前提供"将发送内容预览"。

# 主题二：主会话续聊（CLI 原生 resume）

## 背景

App 关闭后，主会话 PTY 进程被 kill，xterm 缓冲区在内存中也随之消失。重启 App 后用户面对的是一个空终端，之前与 Claude / Codex 的对话上下文从可见层和 CLI 层一起丢失。

实际上 Claude Code 与 Codex 各自都在本地保留了完整对话历史：

- Claude Code：`claude --continue` / `claude --resume <sessionId>`，默认按 cwd 过滤。
- Codex：`codex resume --last` / `codex resume <sessionId>`，默认按 cwd 过滤。

因此恢复机制不需要我们自己存 PTY raw 输出，只要在合适时机给 CLI 传对参数。

## 目标

- 在主终端启动入口提供"新会话 / 继续上次"显式二选一。
- 命中"继续上次"时，按当前选中的 CLI（claude / codex）调用其原生续聊参数。
- 续聊失败时不静默回退，而是回到入口让用户重新选择，避免"以为续上了实际开了新会话"的错觉。
- 续聊时跳过既有的系统 prompt 与 initialUserMessage 注入逻辑，让 CLI 自带的会话历史保持纯净。

## 非目标

- 不自己持久化 PTY raw 输出，不复刻"视觉回放"。
- 不支持跨 CLI 续聊（从 claude 切到 codex 或反向）：切了 CLI 走新会话。
- v1 不处理"同一项目多 PTY 窗口同时续聊"的冲突；先把单会话做稳。
- 不引入第三套 AI 配置，仍复用现有主会话 AI 设置。
- 不改 CLI 自己的会话存储方式，不读 `~/.claude/projects/` 或 `~/.codex/sessions/` 的内部文件。

## 方案概览

UX 形态采用 **A2-α**：App 启动后主终端区域**不自动 spawn PTY**，先显示引导面板，两个大按钮：

```
┌─────────────────────────────────────┐
│       选择本次主会话启动方式         │
│                                     │
│   [ 新会话 ]      [ 继续上次 ]      │
│                                     │
│   当前 CLI: claude（来自 AI 设置）  │
└─────────────────────────────────────┘
```

- 点"新会话" → 走现有 spawn 路径，行为完全不变。
- 点"继续上次" → 用 `--continue` / `resume --last`，并跳过系统 prompt 注入。
- 续聊失败（CLI 立即退出且 exit code 非 0，或在 N 秒窗口内输出含 "no conversation found" 等关键字）→ 回到引导面板并附错误提示。

## 设计一：启动入口 UI

### 1. 组件位置

- 新增 `src/components/MainBootGate.tsx`：放在原本 `MainPanel` 渲染处之上。
- 状态机三态：`idle`（显示引导面板） / `running`（已 spawn，渲染原 `MainPanel`） / `failed`（显示错误提示 + 重新选择按钮）。
- App 启动 / 用户主动"重置主会话"时回到 `idle`。

### 2. 顶栏按钮的持久存在

引导面板只在 `idle` 出现一次；进入 `running` 后，顶栏新增一个**"重置主会话"**入口（与现有"模板"、"Skill 学习"按钮并列），点击：

1. 二次确认弹框 `"当前主会话将被结束，是否继续？"`
2. 确认后 kill 当前 PTY，状态回到 `idle`，再次显示引导面板。

不在 `running` 状态下额外显示"新会话 / 继续上次"按钮，避免误触。

### 3. 当前 CLI 类型可见

引导面板底部一行小字注明：**当前 CLI: claude / codex（来自主会话 AI 设置）**，并附跳转到 `AI 设置` 的链接。换 CLI 后回到引导面板需要重新点。

## 设计二：主进程改造

### 1. `cc:spawn` IPC 协议扩展

`SpawnRequest` 增加一个字段：

```ts
mode: 'new' | 'resume'
```

`ptyManager` 在收到请求时：

- `mode === 'new'`：完全沿用现有逻辑（args、系统 prompt 注入、initialUserMessage）。
- `mode === 'resume'`：
  - 把 args 改成对应 CLI 的续聊形式（见下文）；
  - **跳过** `setTimeout(... injectSystemPrompt ...)` 那一整段；
  - 仍走相同的 PTY 生命周期管理（exit 监听、resize、broadcast）。

### 2. args 转换

新增 `electron/cc/resumeArgs.ts`，纯函数：

```ts
export function buildResumeArgs(
  command: 'claude' | 'codex',
  originalArgs: string[]
): string[] {
  if (command === 'claude') {
    return ['--continue', ...filterConflictingArgs(originalArgs, ['--continue', '-c', '--resume', '-r'])]
  }
  // codex 的 resume 是子命令
  return ['resume', '--last', ...filterConflictingArgs(originalArgs, ['resume', 'fork'])]
}
```

- `filterConflictingArgs` 保证不会重复传同类参数（如用户在 AI 设置里手动加了 `-c`）。
- 注意 codex 的 `resume` 是子命令而不是 flag，要放在最前面。
- 不动 `--model` / `--effort` 等非冲突参数。

### 3. 续聊失败检测

CLI 进程在 N 秒内退出（默认 5s）且 exit code != 0，视为续聊失败。

`ptyManager` 在 `mode === 'resume'` 时：

```ts
const failGuard = setTimeout(() => { /* mark as healthy after 5s */ }, 5000)
proc.on('exit', ({ exitCode }) => {
  if (!healthy && exitCode !== 0 && mode === 'resume') {
    broadcast('cc:resume-failed', { sessionId, exitCode, tail: lastChunk })
  }
})
```

渲染进程收到 `cc:resume-failed` → 回到 `idle` 状态，并把 `tail`（最后 ~1KB 输出）作为错误详情可展开展示。

不做自动重启，由用户决定下一步。

### 4. CLI 是否登录、PTY 是否可用，沿用现有错误路径

`PtyCCProcess.start()` 抛出的异常（找不到 binary 等）路径不动，仍走 `cc:spawn` 的 `{ ok: false, error }` 返回。

## 设计三：渲染进程集成

### 1. `MainBootGate` 实现要点

```tsx
type Phase =
  | { kind: 'idle' }
  | { kind: 'spawning'; mode: 'new' | 'resume' }
  | { kind: 'running'; sessionId: string }
  | { kind: 'failed'; reason: string; tail?: string }
```

- 点"新会话"或"继续上次" → `spawning` → 调 `window.api.cc.spawn({ ..., mode })`。
- 收到 `cc:resume-failed` → `failed`。
- 收到 `cc:spawned` ack → `running`。
- 顶栏"重置主会话"点击 → kill + 回 `idle`。

### 2. 与现有 `App.tsx` 的接合

`App.tsx` 当前在挂载时直接渲染 `MainPanel`。改成：

```tsx
{bootPhase.kind === 'running'
  ? <MainPanel sessionId={bootPhase.sessionId} />
  : <MainBootGate phase={bootPhase} onChoose={...} onReset={...} />}
```

`MainPanel` 内部不动，挂载时仍按 props 中的 `sessionId` 订阅数据流。

### 3. 项目切换 / 窗口刷新

- 切换项目：当前已有"重启主会话"逻辑（kill + 新 spawn）。改造为 kill + 回 `idle`，强制用户再次选择。这样跨项目时也能选续聊。
- 窗口刷新（Cmd+R / Ctrl+R）：渲染进程状态重置，但主进程 PTY 仍在跑。已有保活机制不动；新增逻辑：刷新后 renderer 询问主进程是否已有 session，有就直接进 `running`，没有就 `idle`。

## 设计四：与 AI 设置 / 多 CLI 的关系

- AI 设置中切换主会话 CLI（claude ↔ codex）→ 触发"重置主会话"流程（如果当前在 `running`），自动 kill + 回 `idle`。
- 不在 AI 设置里增加"自动续聊"开关；A2 显式入口已经表达意图。

## 测试设计

### 1. `resumeArgs` 单元测试

### 2. `ptyManager` 集成测试（mock PtyCCProcess）

- `mode: 'resume'` → 不调用 `injectSystemPrompt`。
- `mode: 'resume'` 且 proc 在 5s 内 exit != 0 → 广播 `cc:resume-failed`。
- `mode: 'resume'` 且 proc 5s 内未退出 → 不广播失败，视为成功。

### 3. `MainBootGate` 组件测试

2. `cc:spawn` 协议扩展 `mode` 字段，preload 类型同步。
3. `ptyManager` 按 `mode` 分支 args 与跳过注入逻辑。

10. README 补充"重启后如何续聊"。

每完成一步在终端简述变更。

## 风险与取舍

### 1. CLI 续聊行为可能不完全直观

`claude --continue` 会回放最近一次会话，但具体渲染（是否打印历史消息、是否走"摘要展示"）取决于 CLI 版本。用户可能期望"看起来跟关 app 之前完全一样"，但实际只是"对话上下文续上、视觉布局由 CLI 自己决定"。在引导面板上小字注明"续聊将由 CLI 自身回放历史，显示可能与上次不完全一致"即可。

### 2. 多 PTY 同项目并发

v1 不做冲突检测，两个同项目窗口都点"继续上次"会续到同一份历史，CLI 内部锁机制各异，可能产生竞争。文档上写明"建议每个项目同时只开一个主会话"。

### 3. CLI 版本兼容

`claude --continue` 与 `codex resume --last` 是当前版本可用的稳定选项。后续若 CLI 改名或行为变化，`resumeArgs.ts` 是唯一改动点，便于回归。

### 4. 续聊不等于"项目元数据更新"

续聊跳过了 `buildSystemPrompt`，因此项目重命名、方案路径变更等元数据更新 CLI 不会感知。v1 接受这个限制；用户若需要让 CLI 重新读最新方案，应主动"新会话"。引导面板小字提示这一点。

### 5. 与 FORCE_COLOR 修复的关系

主题一的 `FORCE_COLOR=3` 注入在 `PtyCCProcess.ts` 内统一处理，所有 `mode` 都生效，不需要为本主题特别处理。

## 最终结论

主会话改为显式"新会话 / 继续上次"二选一入口，复用 Claude Code 与 Codex 原生续聊参数（`--continue` / `resume --last`），不自建 PTY 持久化层。续聊时跳过系统 prompt 注入，让 CLI 自带的对话历史保持纯净；续聊失败明确报错回到入口，由用户决定下一步。所有改动集中在 `resumeArgs.ts`、`ptyManager.ts`、`MainBootGate.tsx` 三处主要文件，对现有"新会话"路径零侵入。

# 二期：习惯采集 → 多步 Skill（Phase A）

## 方向调整原因

一期把"候选"产出做成了"prompt 模板"，需要用户主动打开顶栏「模板」→ 选 → 注入，摩擦比预期大。用户实际想要的是：

> 采集到的"经常性操作"被自动提炼成 **Skill**，后续可以**直接调用**这个 Skill 来执行。

二期把"候选 → 模板"链路改造为"候选 → 多步 Skill → 主会话快捷调用"。

## 范围（Phase A）

- 只动 App 内信号；OS 级采集（活动窗口 / shell history 等）留给 Phase B
- 不做键鼠 keylog
- 不写 `~/.claude/skills/` 原生 SKILL.md 文件；Skill 是平台内单元
- 沿用一期的采集层、聚合层、Skill Studio 入口

## 数据模型

新增 `skills` 表（与 `skill_candidates` 并存；candidate 采纳后写入 `skills`）：

```sql
CREATE TABLE IF NOT EXISTS skills (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL,
  name          TEXT NOT NULL,
  description   TEXT,
  trigger       TEXT,          -- 主会话输入条里的触发关键词
  steps         TEXT NOT NULL, -- JSON Step[]
  source        TEXT,          -- 'manual' | 'candidate' | 'imported'
  candidate_id  INTEGER,       -- 反向追踪，可空
  last_used_at  INTEGER
);
CREATE INDEX IF NOT EXISTS idx_skills_trigger ON skills(trigger);
CREATE INDEX IF NOT EXISTS idx_skills_last_used_at ON skills(last_used_at DESC);
```

## Step 类型（v1）

只支持两种类型，覆盖大多数重复操作：

```ts
type Step =
  | { type: 'prompt'; text: string }                 // 发一段提示给主会话
  | { type: 'wait-response'; timeoutMs?: number }    // 等 AI 响应到达静默期
```

`prompt.text` 支持 `{变量名}` 占位符；执行前用户在弹窗填值。后续 Phase 可加 `screenshot` / `pause` / `script`，结构都对齐。

## 主会话输入条（顶部窄条）

主面板（MainPanel）标题栏与 PTY 之间，加一条 `SkillBar`：

- 左侧 `/` 提示图标 + 单行 input
- 聚焦或键入 `/` 时弹 autocomplete：按 `trigger` 前缀匹配 + 按 `last_used_at` 排序
- 选中后：含 `{变量}` → 弹"参数填写"对话框；否则直接执行
- 不影响 PTY 输入路径（PTY 自己的键盘输入仍直达终端）

## Skill 执行引擎

新增 `electron/habit/skillRunner.ts`：

```ts
async function runSkill(sessionId: string, skill: Skill, vars: Record<string,string>): Promise<RunOutcome>
```

实现：
1. 对每个 step 按类型派发
2. `prompt`：替换 `{var}` 后通过 `cc.sendUser` 发送
3. `wait-response`：监听 `cc:data` 直到静默 N 秒或超时
4. 记录 `last_used_at`；失败返回错误不阻塞后续 step

## 生成器升级

`buildGenerationPrompt(cluster)` 要求 CLI 返回新结构：

```json
{
  "name": "...",
  "description": "...",
  "trigger": "...",

  "variables": [...],
  "rationale": "..."

```

`parseGenerationResponse` 同步调整；本地启发式兜底也升级输出。

## UI

- 现有 `SkillCandidatesPanel` 改造为支持 step 列表预览与编辑
- 新增 `SkillLibraryPanel`：列出已采纳 skills，可编辑/删除/手动新建/试运行

# 三期：项目知识库（Project KB）

## 方向

软件随用户使用某个项目的频次，**渐进式积累对项目的结构化理解**，存为本地知识库；新启动会话时 AI 能拿到这份理解（digest 注入 + MCP 查询工具），不需要每次重新解释项目背景。

## 范围（Phase 1）

- 仅项目独立：每个 repo 一份 `kb.db`
- 不打断主 AICLI：所有 AI 调用走独立子进程（沿用 habit generator 同款）
- AI 触达：digest 注入（通用，Claude+Codex 都吃）+ MCP 查询工具（暂仅 Claude Code 生效）
- 数据源全部来自已有信号：git log / 文件 mtime / `habit_events` 里的用户 prompts / 既有 KB
- 不额外采集任何 OS 级信号
- 不做跨项目联合检索（KB 本身留好扩展位）

## 数据模型

```
<repo>/.multi-ai-code/project-kb/
  ├── kb.db           -- SQLite，含 FTS5 索引
  └── digest.md       -- 派生：≤ 2 KB 紧凑总览，每次更新自动重生
```

```sql
CREATE TABLE kb_entries (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL,
  topic         TEXT NOT NULL,         -- "认证系统" / "diff 渲染"
  summary       TEXT NOT NULL,         -- 1-3 句话核心信息
  evidence      TEXT,                  -- JSON: { commits: [...], files: [...], prompt_ids: [...] }
  importance    REAL NOT NULL DEFAULT 0.5,
  tier          TEXT NOT NULL DEFAULT 'hot',  -- 'hot' | 'warm' | 'cold' | 'pinned'
  access_count  INTEGER NOT NULL DEFAULT 0,
  last_accessed_at INTEGER
);
CREATE INDEX idx_kb_topic    ON kb_entries(topic);
CREATE INDEX idx_kb_tier     ON kb_entries(tier);
CREATE INDEX idx_kb_updated  ON kb_entries(updated_at DESC);
CREATE VIRTUAL TABLE kb_fts USING fts5(topic, summary, content='kb_entries', content_rowid='id');
```

## 智能调度

平台主进程里跑独立的 KB scheduler，每 60 秒轻量 tick：

**进入条件（任一）**：
- 用户连续 15 分钟没新 prompt（空闲间隙）
- 自上次总结以来累计 ≥ 5 条新 prompt
- 距上次总结 ≥ 24 小时且至少 1 条新 prompt（兜底）

**强制跳过（即使进入条件满足）**：
- 主会话 PTY 最近 10 秒还在收 AI 数据（流式中）
- 距上次总结不足 30 分钟（速率限）
- 主会话最近 30 秒提交过 prompt（用户还在等回答）
- 没有 CLI 配置可用

调度状态字段：
- `lastSummaryAt`：最近一次成功完成时间
- `lastAiActivityAt`：最近一次 cc:data 收到的时间
- `lastUserPromptAt`：最近一次用户提交 prompt 的时间
- `pendingSignalCount`：自 lastSummaryAt 起新增的 ai_prompt_main 事件数

所有判断是**纯函数**，单独抽 `shouldRunSummary({signals}, now)` 便于测试。

## 不打断主 AICLI

KB 的所有 AI 调用（生成 + 压缩）都走独立子进程：
- 沿用现有 `runCliGeneration(prompt, settings)` 工具
- 共享一个 **CLI 串行队列**（habit generator 也接入）：habit 和 KB 同一时刻最多跑一个子进程，避免 OAuth 冲突或 rate-limit

队列接口：
```ts
enqueueCliJob<T>(label: string, run: () => Promise<T>): Promise<T>
```

队列内部 FIFO，按 label 可见调试。

## 数据源

每次摘要 tick 跑前读取：
- `git log -50 --pretty=format:%h|%s` （最近 50 条 commit）
- `find src -type f -newer <30天前> -printf` 等价（最近修改过的源码文件 mtime + 路径）
- `listHabitEventsSince(lastSummaryAt)` 过滤 kind=`ai_prompt_main` 取用户最近 prompts
- 既有 `kb.db` 的 hot tier 全部 + warm tier topic 列表

不读 AI 回复、不读文件内容、不读 secrets。

## 摘要生成

构造一段紧凑提示给子进程 CLI，要求返回结构化 JSON：

```json
{
  "topics": [
    { "topic": "...", "summary": "...", "evidence": {...}, "importance": 0.0-1.0 }
  ],
  "digest": "本项目是 ... 当前焦点 ... 最近关注 ..."
}
```

写入 KB 时：
- topic 若已存在 → 增量更新 summary，increment access_count
- topic 新 → 插入到 hot tier
- digest.md 全量覆盖（这是派生产物）

## 三段控量

每次 summary 跑完，自动检查 tier 上限：

| Tier | 上限 | 溢出策略 |
|---|---|---|
| hot | 30 条 | 按 importance 取末位若干转 warm（AI 把它们合并成主题级段落） |
| warm | 30 个主题段 | 30 天没被命中查询 → 进 cold |
| cold | 5 段 | 总文件 > 500 KB → 强制 AI 再压一轮，cold 段数减半 |
| pinned | 不限 | 永不降级 |

降级用 AI 子进程做（"把这 N 条合成 1 段段落，保留关键事实"），失败就跳过本轮。

**重要度评分**：
```
score = exp(-age_days / 14) × log(1 + access_count) × (pinned ? 999 : 1)
```

## 查询接口

主进程暴露给 IPC + MCP 共用：

```ts
queryKb(repoRoot, q, opts?): KbEntry[]      // FTS5 LIKE + 相关度
listKbByTopic(repoRoot, topic): KbEntry[]
listKbRecent(repoRoot, n): KbEntry[]

listKbTopics(repoRoot): string[]
pinKb(repoRoot, id) / unpinKb(repoRoot, id)
deleteKb(repoRoot, id)
updateKb(repoRoot, id, patch)                // 让用户能手动改 summary
forceCompact(repoRoot)                       // 手动触发一次压缩

```

  - 选项 A：MCP 用纯 JS sqlite 实现（如 `sql.js`，较慢但兼容）
  - 选项 B：构建时打包两份 better-sqlite3 binary（Electron 用 + 纯 Node 用）
  - v1 先尝试方案 A；性能不行再切 B。

# 四期：屏幕采集 → 习惯学习信号

## 方向

把 **屏幕画面 / 活动窗口** 作为新的 habit_events 信号源，让 habit 聚类 + 项目 KB 摘要不仅看用户键入的 prompt，还能感知"用户是怎么在自己电脑上工作的"。

定位：**实时被动监控**（用户主动选了这个方向），但**分采样**——不录视频，只采"够用即可"的极轻信号。

## 范围（Phase 1）

**做**：
- **L1**：每 5s 采一次前台窗口名 + App/进程名（用 `active-win` 库）
- **L2**：每 30s 采一次主屏缩略图 256×144 webp，写盘 + habit_events 行
- 并入现有"习惯监控"对话框（新增 tab），不开新独立 UI
- 同时写入 `habit_events`（跨项目）**与** 对应 target_repo 的 KB summarizer 输入

**不做**：
- L3 事件触发短片
- L4 全屏录像 / OBS / libobs
- 实时 OCR / 实时 LLM 分析（按已有 24h 调度，分析与采集解耦）
- 跨监视器（只采主屏）

## 数据模型

复用 `habit_events` 表，加两个新 kind：

```
'screen_window' — payload: { title, appName, bundleId?, processId? }
'screen_frame'  — payload: { framePath, w, h, app? }
```

L2 缩略图存盘：
```
<rootDir>/screen-samples/<YYYY-MM-DD>/<ISO-timestamp>.webp
```

retention sweep（已有，主进程定时）扩展：
- 删 habit_events 超期行时，同时删对应的 webp 文件
- 总盘上限 1 GB；超过先按 mtime 清最旧到 800 MB

## 采集器（`electron/habit/screenSampler.ts`）

```ts
startScreenSampler({
  l1IntervalMs: 5000,
  l2IntervalMs: 30000,
  paused: () => boolean,        // 顶栏按钮读这个
  appBlocklist: () => string[], // 设置面板读这个
  onWindow: (info) => void,     // 写 habit_events kind='screen_window'
  onFrame: (info) => void,      // 写盘 + habit_events kind='screen_frame'
})
```

- 两个 setInterval；setInterval 用 `unref()` 避免阻塞 quit
- 拿前台窗口 → 标题/进程名过 blocklist 正则；命中跳过本轮
- 缩略图：`desktopCapturer.getSources({types:['screen'], thumbnailSize:{w:256,h:144}})` → NativeImage.toPNG() → 用 `sharp` 转 webp（已是 devDependencies）；若 sharp 不可用，回退 PNG
- 顶栏指示灯由 `paused()` 决定颜色

## 隐私护栏（default-on）

| 项 | 默认 | 实现 |
|---|---|---|
| 顶栏指示灯 | 永远显示 | App.tsx 加一个小圆点 |
| 一键暂停 | 永远可点 | 点击切 `paused` flag |
| App 黑名单 | 默认含 1Password/Bitwarden/浏览器隐私模式 | 设置面板可改；窗口标题 / appName 命中 = 不采集本轮 |
| secret 正则过滤 | 默认开 | 入库前对 window_title 做 `sk-\S+` / `Bearer \S+` / 长 base64 串置 `<redacted>` |
| 本地加密 | 关（v2） | 留扩展位 |

## 信号去向

- **永远入 habit_events**（跨项目可见的"我电脑在干什么"行为流）
- **同时入 KB summarizer 输入**，前提：当前前台窗口能识别出已知 target_repo
  - 启发式识别：terminal cwd 匹配 / IDE 标题里含 repo basename / VS Code window title 末尾 `[repo-name]`
  - 识别失败 → 仅入 habit，跳过 KB
- 24h 内同 `window_title` 在 habit_events 中去重

## 实施顺序

1. `active-win` 依赖加进来
2. `electron/habit/screenSampler.ts` + 单元测试（mock active-win + mock desktopCapturer）
3. retention sweep 加 webp 文件清理 + 1 GB 上限
4. 顶栏指示灯 + 一键暂停按钮
5. HabitMonitor 对话框加 "屏幕采集" tab：开关 L1 / L2、保留期、App 黑名单、最近缩略图浏览
6. 信号接 aggregator + KB summarizer
7. macOS Screen Recording 权限引导（首次启动检测）
8. README 补节 + 端到端联调

## 非目标（明确）

- 不录 mp4/webm 视频
- 不抓 keystroke / mouse log
- 不上传任何采样到云端（除非走 KB summarizer 既有的"最小集"路径）
- 不实时 OCR（v2 看效果再决定）
- macOS 任务栏的"屏幕被采集"图标无法关闭（OS 强制）—— 视为"用户被告知中"

## 风险与取舍

- **macOS 权限**：首启需要用户在系统设置授权"屏幕录制"，否则 desktopCapturer 返回空
- **杀软误报**：Windows 上某些 EDR 可能拦截 desktopCapturer 首次调用，放行后即可
- **CPU 峰值**：每 30s 截图 + 缩放 ~100-300 ms。日常无感，但要监控
- **L2 文件爆炸**：30 天 ~6 万张 webp。1 GB 上限是硬约束，超过靠 retention sweep 兜底
- **窗口标题敏感**：典型如"密码 - Password Vault"。靠 App 黑名单 + secret 正则双重防

# Desktop IM 本地消息库（历史消息持久化）

## 背景与现状

Desktop IM（qt-im）当前的历史消息完全依赖腾讯 IM SDK 的漫游拉取（`messagesReceived`），SDK 只保留最近几条，重登后历史大量丢失。代码考古结论：

- `LocalChatHistoryStore`（JSON 文件版，save/load 整个 ChatState）**已实现且有单测，但从未接线**——应用层无任何调用方
- 实时消息（`receiveText/Image/Voice/File`）只进内存 `ChatState`
- `ChatState::appendMessageForRestore` **无按 id 去重**，直接 append

## 目标

- 所有通讯消息（收/发/图片/语音/文件引用/状态）持久化到本地 **SQLite 数据库**（每账号一个库）
- 登录后**先从数据库加载全部历史**展示到消息列表；SDK 漫游拉取降级为**补充**（按消息 id 去重合并）
- 实时收发即时落库；消息状态（发送中/成功/失败）变更同步更新
- 删除好友时级联删除该 peer 的本地消息（与现有 `removeContactAndMessages` 行为一致）

## 非目标

- 不做云端同步 / 多设备合并
- 不做消息搜索（表结构留索引位，后续可加 FTS）
- v1 不设消息量上限 / 自动清理（留清理接口）
- 不迁移旧 JSON 数据（该 store 从未接线，无存量数据）

## 方案

### 1. 存储层：`LocalMessageDatabase`（新，SQLite）

- 依赖：`find_package(Qt5 ... Sql)` + `Qt5::Sql`（windeployqt 自动带 Qt5Sql.dll + sqlite 插件，绿色包无额外负担）
- 库文件：`<rootDir>/<ownerUserId>/messages.db`（rootDir 沿用现有媒体存储同级目录）
- 表结构：

```sql
CREATE TABLE IF NOT EXISTS contacts (
  user_id      TEXT PRIMARY KEY,
  display_name TEXT NOT NULL
);
CREATE TABLE IF NOT EXISTS messages (
  id            TEXT PRIMARY KEY,          -- 消息 id，天然去重
  from_user     TEXT NOT NULL,
  to_user       TEXT NOT NULL,
  peer          TEXT NOT NULL,             -- 冗余列：会话对端，加速按会话查询
  direction     INTEGER NOT NULL,          -- 0 incoming / 1 outgoing
  status        INTEGER NOT NULL,
  text          TEXT NOT NULL DEFAULT '',
  created_at    INTEGER NOT NULL,          -- millis
  has_image     INTEGER NOT NULL DEFAULT 0,
  image_path    TEXT, image_w INTEGER, image_h INTEGER, image_bytes INTEGER,
  has_voice     INTEGER NOT NULL DEFAULT 0,
  voice_path    TEXT, voice_seconds INTEGER,
  has_file      INTEGER NOT NULL DEFAULT 0,
  file_path     TEXT, file_name TEXT, file_mime TEXT, file_bytes INTEGER
);
CREATE INDEX IF NOT EXISTS idx_messages_peer_time ON messages(peer, created_at);
```

- 接口（增量写，不做全量重写）：
  - `loadAll(ChatState&)`：启动加载，联系人 + 全部消息按 created_at 升序
  - `upsertContact / removeContactCascade`（级联删该 peer 消息）
  - `insertMessageIfAbsent(msg)`：`INSERT OR IGNORE`，返回是否新插入（供漫游合并去重复用）
  - `updateMessageStatus(id, status)`

### 2. 接线：`RemoteIMApplication`

- 构造注入 `LocalMessageDatabase`（可空指针，测试可不带库跑）
- 构造后立即 `loadAll` → `stateChanged`（**登录前本地历史即已可见**）
- `messagesReceived`（SDK 漫游，几条）：逐条 `insertMessageIfAbsent`，仅新消息才 `appendMessageForRestore`（同时给 ChatState 补按 id 去重）
- 实时收发/状态回执：落库 + 更新状态
- `deleteContact` → `removeContactCascade`

### 3. ChatState 排序与去重

- `appendMessageForRestore` 增加按 id 去重
- 加载 + 漫游合并后消息按 `createdAtMillis` 稳定排序（展示顺序与时间一致）

### 4. 旧 JSON store 处置

qt-im C++ 版 `LocalChatHistoryStore` 确认为**纯死代码**（除自身单测外零调用方），随本次改造**直接删除**（源文件 + 测试 + CMake 条目）。注意：Android（`LocalChatHistoryStore.java`）与 iOS 端的同名实现**已实际接线**（MainActivity/RemoteIMSessionController 在用），属于各自平台的独立实现，不受影响、不动。

## 测试

- `LocalMessageDatabaseTest`：建库建表、插入/去重（同 id 二次插入不生效）、按会话排序加载、状态更新、级联删除、重开库数据仍在
- `RemoteIMApplicationTest` 扩展（FakeRemoteIMClient）：启动即加载历史；漫游消息与库中重复 → 不重复展示；实时消息落库后重启（重建 app 对象）可恢复
- 手工验证：登录 whq-iphone → 收若干消息 → 退出重登 → 历史完整；SDK 漫游重复消息不重复显示

## 实施顺序

1. CMake 加 Qt5 Sql 组件；`LocalMessageDatabase` + 单测
2. `ChatState::appendMessageForRestore` 去重 + 排序 + 单测
3. `RemoteIMApplication` 接线（加载/合并/落库/级联删）+ 单测
4. MainWindow 无需改动（消息列表已由 stateChanged 驱动）
5. 打包验证 Qt5Sql.dll 随包；手工端到端验证
6. 提交推送 + 更新 Release

# Electron IM 模块：漫游补充拉取（本地消息库补全）

## 现状盘点（与 qt-im 的差异）

Electron 端 IM **已具备**大部分基础：
- SQLite 持久化：`electron/remote-im/messageStore.ts`（`remote_im_messages` 表，含 `remote_message_id` 列 + 索引）
- 入库去重：`create()` 已按 `(provider, remote_message_id)` 查重（入站消息带 TIM 消息 id）
- 启动加载：`App.tsx` 经 `listMessages(projectId, 100)` 从库读历史

**缺失**：
1. **登录后无 SDK 漫游补充拉取**——渲染端 `tencentImClient.ts` 只监听 `MESSAGE_RECEIVED`（在线实时推送），Electron 离线期间对方发的消息永远进不了库，重启后彻底缺失
2. **出站消息未回填 `remote_message_id`**——发送成功后行里仍是 NULL，漫游重投自己发过的消息时无法去重、会出现重复
3. 历史加载硬编码 100 条上限

## 方案

### 1. 漫游补充拉取（核心）
- `tencentImClient.ts`：SDK_READY 后调 `chat.getConversationList()` → 对每个 C2C 会话 `chat.getMessageList({ conversationID })` 拉最近漫游
- 转换为与实时路径同构的 incoming/outgoing 输入（携带 `remoteMessageId = msg.ID`、`createdAt = msg.time*1000`），经既有 `create()` 去重入库
- 完成后通知渲染端刷新消息列表（复用现有 messages 刷新事件）

### 2. 出站回填 remote id
- `UpdateRemoteImMessageStatusInput` 增加可选 `remoteMessageId`；TIM `sendMessage` resolve 的 message.ID 在标记 sent 时一并写回
- 漫游重投自己发的消息即可命中去重

### 3. 加载上限
- `listMessages` 上限 100 → 500（内容多为 AICLI 长文，暂不做无限加载/分页，留后续）

### 4. 测试
- messageStore：出站回填 remote id 后按 id 去重的用例
- tencentImClient：漫游拉取→转换→入库调用的单测（mock chat SDK）
- 手工：Electron 离线期间从手机发消息 → 启动 Electron → 消息出现且不重复

## 附带发现（qt-im 同类隐患，另行小修）
qt-im 出站消息同样未采用 SDK 返回的消息 id（本地 UUID），自己发的消息在重启后被漫游重投时会重复。修法一致：发送回调中取 SDK 消息 id 回填。
