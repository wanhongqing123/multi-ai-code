# 习惯学习智能体（Skill 自动建议）设计

## 背景

当前 Multi-AI Code 已有两个产品级 AI 角色：

- 主会话 AI：驱动主界面 PTY 终端
- 仓库查看分析 AI：在 `仓库查看` 窗口里做代码标注分析

用户在长期使用过程中，会重复输入大量相似的 prompt，重复触发相似的工作流。这些重复模式当前需要用户自己手动整理到顶栏的"模板"功能里，门槛高、容易遗忘。

本次新增**第三个产品级 AI 角色：习惯学习智能体（Habit Agent）**，定位是被动观察 + 定期汇总 + 主动建议：

- 被动观察：在用户操作过程中静默采集 App 内行为信号
- 定期汇总：按周期对采集到的行为做本地聚合分析
- 主动建议：把聚合结果交给 AI CLI，生成"候选 prompt 模板"，由用户决定是否采纳进现有"模板"系统

## 目标

- 新增"习惯学习智能体"作为第三个独立 AI 角色，与主会话 AI、仓库查看分析 AI 并列。
- 仅在本 App 内部采集行为信号，不监控本机操作系统其他活动。
- 行为数据默认只存本地，仅在 skill 生成时把**最小聚合摘要**发给 CLI。
- 定期（每天/每周）汇总，生成"候选 skill"列表，由用户决定采纳/编辑/丢弃。
- 采纳后的 skill 直接进入现有顶栏"模板"系统，复用现有调用入口。
- 提供清晰的开关、采集源管理、保留期管理、清空数据入口。

## 非目标

- 不监控系统级行为（不抓键盘、鼠标、窗口活动、其他进程）。
- 不读取 shell 历史、git 提交、本机其他目录的文件变动。
- 不实时弹窗打断用户。
- 不自动把生成的 skill 直接执行或注入会话；所有 skill 必须用户显式采纳后才进入模板列表。
- 不为该智能体设计独立的对话窗口；它的产物只有"候选 skill 列表"。
- 不做云同步、不做跨设备数据迁移（行为数据始终本地）。
- v1 不为本地数据库引入 SQLCipher 等应用层加密；只依赖操作系统盘加密。如果用户需要更强保护，需在后续版本单独立项。
- 不重写现有"模板"系统的数据结构；保持 `{id, name, body}` 三字段兼容。

## 方案概览

本方案分四个子系统：

1. **信号采集层（Collector）**：在 App 内现有关键交互点静默记录事件到本地 SQLite。
2. **本地聚合层（Aggregator）**：周期性对原始事件做无 LLM 的聚类与打分。
3. **生成层（Generator）**：把聚类摘要交给 CLI，生成候选 prompt 模板。
4. **审阅 UI**：候选 skill 列表 + 采集设置 + 数据查看与清空入口。

### 数据流

```
用户操作 → Collector 钩子 → 本地 SQLite (habit_events 表)
                                  │
                                  ▼
            周期调度（每天/手动触发）
                                  │
                                  ▼
       Aggregator（纯本地）：聚类 + 打分 → top-K 候选簇
                                  │
                                  ▼
       Generator：构造最小摘要 → 交给习惯学习 AI CLI
                                  │
                                  ▼
        CLI 返回 {title, body, meta} → 写入 skill_candidates 表
                                  │
                                  ▼
            审阅 UI（用户决策：采纳 / 编辑 / 丢弃 / 暂不处理）
                                  │
                                  ▼
                采纳 → 写入现有 localStorage 模板系统
```

## 设计一：信号采集层

### 1. 采集范围（仅 App 内）

| 事件类型 | 来源代码 | 记录字段（payload） |
|---|---|---|
| `pty_cmd` | `electron/cc/PtyCCProcess.ts` | 用户在主会话 PTY 中提交的命令行 |
| `ai_prompt_main` | 主会话提交点 | 发送给主会话 AI 的 prompt 原文 |
| `ai_prompt_repo` | `electron/repo-view/repoAnalysisManager.ts` | 仓库查看窗口提交的分析 prompt |
| `diff_annotation` | `src/components/DiffViewerDialog.tsx` | Diff 批注文本 |
| `repo_view_annotation` | `src/repo-view/AnalysisPanel.tsx` | 仓库查看里的代码标注文本 |
| `template_used` | `src/components/TemplatesDialog.tsx` 注入入口 | 被调用的模板 id 与 name |
| `plan_imported` | `electron/orchestrator/plans.ts` | 外部方案导入的源类型（仅类型，不含路径） |

不采集：
- 终端输出内容
- AI 返回的文本
- 文件内容
- 完整的方案文件正文

理由：采集 input 侧已经能描述"用户的意图"，output 侧主要是噪声，且增大数据体积与隐私风险。

### 2. 存储

新增 SQLite 表（写入现有 `electron/store/db.ts` 维护的 db 文件）：

```sql
CREATE TABLE IF NOT EXISTS habit_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts INTEGER NOT NULL,
  kind TEXT NOT NULL,
  payload TEXT NOT NULL,
  project_id TEXT,
  repo_path TEXT,
  source_window TEXT
);
CREATE INDEX IF NOT EXISTS idx_habit_events_ts ON habit_events(ts);
CREATE INDEX IF NOT EXISTS idx_habit_events_kind ON habit_events(kind);
```

- `payload` 为 JSON 字符串
- `project_id` / `repo_path` 用于后续按项目维度分组（v1 不强分桶但记录该维度）
- `source_window` 取值如 `main` / `repo-view` / `diff-review`

### 3. 默认开关与保留期

- 主开关默认 **开启**。首次开启会一次性弹出告知面板，说明采集范围、不采集什么、数据去向、如何关闭与清空。告知面板有"我知道了"和"立即关闭采集"两个出口。
- 每个事件类型有独立子开关，默认全开。
- 默认保留期 90 天，超期自动按 ts 删除。
- 用户可在设置面板里：
  - 切主开关
  - 切每个事件类型的子开关
  - 调整保留期（30 / 60 / 90 / 180 天）
  - 查看最近 N 条原始事件
  - 一键清空所有 habit_events 记录

### 4. 采集钩子的实现方式

新增 `electron/habit/collector.ts`，对外暴露：

```ts
recordEvent(kind: HabitEventKind, payload: object, ctx?: { projectId?: string; repoPath?: string; sourceWindow?: string }): void
```

在上述各采集点处调用。调用方不需要关心：

- 主开关是否开启（关闭时函数直接 no-op）
- 子开关是否开启
- DB 是否就绪

把所有判断收敛在 `recordEvent` 内部，最大化降低改动面，避免污染现有业务代码。

## 设计二：本地聚合层

### 1. 调度

新增 `electron/habit/scheduler.ts`：

- App 启动后等待 N 秒（默认 60s）跑一次轻量聚合，把"距上次完整聚合超过 24h"的状态置 dirty
- 每隔 24h 跑一次完整聚合
- 用户可在 UI 手动触发"立即分析"

聚合本身不调 LLM，是纯本地计算，CPU 与磁盘开销可控。

### 2. 聚合算法 v1

新增 `electron/habit/aggregator.ts`，纯函数，便于测试。

输入：最近 30 天的 `habit_events` 列表  
输出：top K 个候选簇，每个含 `{ representativeText, sampleEventIds, size, score, kindDistribution }`

算法：

1. 按 `kind` 分组，避免跨类型聚类（命令行和 prompt 文本天然不同）。
2. 对每组内的文本，做轻量预处理：
   - 去掉路径、行号、临时数字
   - 中英文混合分词（按空白 + 标点切，不强求 NLP 库）
3. 计算 shingles（k=3 词组），用 Jaccard 相似度 ≥ 0.4 做并查集聚类。
4. 簇得分 = `cluster_size * recency_decay`，其中 `recency_decay = exp(-Δdays / 14)`。
5. 过滤 `size < 3` 的簇。
6. 按得分排序，取 top K（默认 10）。

特点：

- 完全本地，无外部依赖
- 算法朴素但可解释、可测试
- 后续可在不破坏接口的前提下替换为 MinHash / TF-IDF / 嵌入向量

### 3. 跨项目维度

v1 不按 `project_id` 强分桶，所有项目的事件进同一个池。但簇结构里附带 `projectDistribution`，让生成层能在提示词里告诉 CLI：

- 这是"跨多项目通用模式"还是"专属某项目的模式"
- 不同模式对应不同的 skill 命名风格（前者更通用，后者更具体）

## 设计三：生成层

### 1. 与现有 AI 配置的关系

**直接复用主会话 AI 配置**，不引入第三套配置。

- 习惯学习智能体每次生成 skill 时，调用的 CLI 与主会话相同
- 用 binary / args / env 也完全沿用主会话设置
- `AI 设置` 对话框保持现状（仍为主会话 + 仓库查看 两套）
- 在习惯学习 UI 内部用一行小字注明：当前使用的 CLI 来自主会话 AI 设置

后续如有需要再拆出独立配置（例如允许用更便宜的模型跑批量分析），届时新增一个"使用独立配置"的开关即可，不破坏当前结构。

### 2. 发送给 CLI 的内容（B 模式：最小化导出）

新增 `electron/habit/generator.ts`。每个候选簇构造**单独一次**对话调用，发送内容严格限定为：

```
[簇描述]
- 类型: ai_prompt_main / pty_cmd / ...
- 出现次数: 12
- 时间跨度: 最近 18 天
- 涉及项目数: 3
- 是否跨项目通用: 是

[代表样本 3-5 条]
1. "帮我看看 src/foo.ts 的实现"
2. "解释 electron/main.ts 的会话管理逻辑"
3. ...

[要求]
基于以上重复模式，生成一个可复用的 prompt 模板。返回严格 JSON：
{ "title": "...", "body": "...", "variables": ["..."], "category": "...", "rationale": "..." }
```

注意：

- 不发完整事件列表，只发 3-5 条代表样本（取簇中心距离最小的几条）
- 不发原始时间戳、不发 project_id 原始值（用"涉及项目数"代替）
- 不发文件内容、不发 AI 返回内容

每次生成失败时记录失败原因，状态置为 `error`，不阻塞其他候选。

### 3. 候选 skill 存储

```sql
CREATE TABLE IF NOT EXISTS skill_candidates (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at INTEGER NOT NULL,
  cluster_kind TEXT NOT NULL,
  cluster_size INTEGER NOT NULL,
  source_event_ids TEXT NOT NULL,
  representative_samples TEXT NOT NULL,
  generated_title TEXT,
  generated_body TEXT,
  generated_meta TEXT,
  status TEXT NOT NULL DEFAULT 'pending',
  reviewed_at INTEGER,
  error_message TEXT
);
```

`status` 取值：`pending` / `accepted` / `edited` / `discarded` / `snoozed` / `error`

## 设计四：审阅 UI

### 1. 顶栏入口

新增按钮：**`Skill 学习`**，位置在顶栏现有 `模板` 旁边。

点击打开 `SkillStudioDialog`，包含两个 tab：

- **候选**：未审阅的 `pending` / `snoozed` 列表
- **采集**：开关、子开关、保留期、最近事件查看、清空

当前所用的 CLI（来自主会话 AI 设置）在对话框顶部以一行小字注明，附跳转到 `AI 设置` 的链接。

### 2. 候选 tab

每条候选展示：

- 生成时间
- 簇大小 + 类型
- AI 生成的 title / body / category / rationale
- 操作按钮：
  - `采纳` → 写入现有 localStorage 模板，status = `accepted`，关掉条目
  - `编辑后采纳` → 打开编辑器修改 title/body，提交后状态 `accepted`
  - `丢弃` → 状态置 `discarded`，从列表移除
  - `暂不处理` → 状态置 `snoozed`，N 天后重新出现
  - `查看证据` → 展开看本簇的代表样本与 source_event_ids 链接

- 在发送给 CLI 之前还有一个**预览发送内容**入口：让用户在生成前能看到本次将发给 CLI 的精确 payload（B 模式承诺的可视化兜底）。

### 3. 采纳后的模板写入

复用现有结构：

```ts
{
  id: `t_skill_${candidateId}`,
  name: generated_title,
  body: generated_body
}
```

由于现有模板存 localStorage、不在主进程，采纳逻辑放在渲染进程：渲染进程读到候选数据后调用现有 `load() / save()` 模式追加，不通过 IPC 复杂回传。

### 4. 采集 tab

包含：

- 主开关（默认关闭）
- 各 kind 子开关（默认全开）
- 保留期下拉
- "最近 100 条事件"只读列表（按 kind 着色，时间倒序）
- "清空所有数据" 按钮（二次确认）

### 5. 首启告知

App 升级到本版本后**首次进入主界面**时弹一次告知面板（因为采集默认开启，需要让用户知情）：

- 我们采集什么（7 个事件类型）
- 不采集什么（无系统级监控、不记 output 侧）
- 数据存哪里（本地 SQLite）
- 什么时候发出去、发什么（生成 skill 时只发最小聚合摘要）
- 可以随时关闭与清空

面板有两个按钮：
- `我知道了` → 维持默认开启状态，写入"已告知"标记
- `先关闭采集` → 关掉主开关，仍写入"已告知"标记，但不再每次启动都弹

告知只弹一次，状态写本地。

## 测试设计

### 1. Collector 测试

- 主开关关闭时 recordEvent 不写入 DB
- 子开关关闭时对应 kind 不写入
- 各种 payload 类型均能正确序列化
- 超出保留期的记录会被定期清理

### 2. Aggregator 测试

- 完全相同的文本会聚成一个簇
- 路径不同但模式相同的 prompt 会聚成一个簇（验证预处理去掉路径的能力）
- 簇大小 < 3 会被过滤
- recency_decay 对得分排序的影响符合预期
- 完全没有事件时返回空列表

### 3. Generator 测试

- 构造的 CLI payload 中**不含**原始 project_id
- payload 中样本数不超过 5 条
- CLI 返回非法 JSON 时候选 status 置 `error`
- CLI 调用失败时记录 error_message，不影响其他候选

### 4. UI 测试

- 候选列表正确展示与排序
- 采纳后的候选从列表移除并写入模板
- 编辑后采纳保留用户修改
- 暂不处理后会在指定天数后重新出现
- 预览发送内容显示与实际生成调用一致

### 5. 隐私不变量测试

- payload 序列化中不出现 `repo_path`、文件内容、AI 返回文本（即使作为子串）
- 主开关关闭后所有采集点都为 no-op
- 清空数据后 DB 表为空

## 实施顺序

1. DB schema + 迁移
2. Collector 接口与主开关/子开关逻辑（无 UI）
3. 在最少干扰的两个采集点（PTY 命令 + Diff 批注）先接入，验证可行
4. Aggregator 纯函数 + 完整测试
5. 采集 tab UI（先让用户能开关、看数据、清空）
6. Scheduler 与"立即分析"按钮
7. Generator + CLI 调用 + 失败处理
8. 候选 tab UI 与采纳流
9. 把剩余采集点接入（AI prompt、仓库查看标注、模板使用、方案导入）
10. 首启引导
11. 集成测试与隐私不变量测试
12. README 补充新角色说明

每完成一步在终端简述变更，按 CLAUDE.md 要求执行。

## 风险与取舍

### 1. 高频 ≠ 值得抽成 skill

朴素频次模型会把"输入回车"这种噪声选成 top 候选。缓解：

- 文本最小长度阈值（默认 ≥ 8 字符）
- 同一文本在 24h 内最多计 1 次
- 排除单词级别短文本（仅 pty_cmd 类）

### 2. 多项目共享 skill 容易产生不通用模板

跨多项目聚类时，如果 3 个项目实际上是 3 套语义不同的代码（前端/后端/桌面），生成的"通用"模板可能两边都不贴合。缓解：

- 在 generator 提示里明确告知 CLI"跨项目数 N、是否同类型"
- 让 CLI 在不确定时主动给"建议拆成两个变体"的提示
- 用户可在编辑后采纳里手工拆分

### 3. 隐私感知

即使是 B 模式，也要让用户在生成前能清楚看到"本次将发什么"。已通过预览发送内容入口缓解。

### 4. CLI 失败长尾

生成调用是异步、批量、可能数十次的过程。失败要：

- 单条失败不阻塞其他候选
- 错误信息保留在 skill_candidates.error_message
- UI 上有"重试本条"入口

### 5. 与现有 localStorage 模板系统耦合

采纳路径直接写 localStorage，绕过主进程 SQLite。这会带来：

- 跨设备同步缺失（与现有模板一致，不算回退）
- 多窗口同时打开时可能有竞争

v1 接受这个一致性级别。后续若把现有模板迁到 SQLite，再统一改造。

## 最终结论

新增"习惯学习智能体"作为第三个产品级 AI 角色。它被动观察 App 内行为、本地周期聚类、按 B 模式（最小化导出）调用 CLI 生成候选 prompt 模板，由用户审阅采纳后流入现有"模板"系统。所有数据默认本地、默认关闭采集、提供完整的开关与清空入口，并在生成前提供"将发送内容预览"。

# 主题二：主会话续聊（CLI 原生 resume）

## 背景

App 关闭后，主会话 PTY 进程被 kill，xterm 缓冲区在内存中也随之消失。重启 App 后用户面对的是一个空终端，之前与 Claude / Codex 的对话上下文从可见层和 CLI 层一起丢失。

实际上 Claude Code 与 Codex 各自都在本地保留了完整对话历史：

- Claude Code：`claude --continue` / `claude --resume <sessionId>`，默认按 cwd 过滤。
- Codex：`codex resume --last` / `codex resume <sessionId>`，默认按 cwd 过滤。

因此恢复机制不需要我们自己存 PTY raw 输出，只要在合适时机给 CLI 传对参数。

## 目标

- 在主终端启动入口提供"新会话 / 继续上次"显式二选一。
- 命中"继续上次"时，按当前选中的 CLI（claude / codex）调用其原生续聊参数。
- 续聊失败时不静默回退，而是回到入口让用户重新选择，避免"以为续上了实际开了新会话"的错觉。
- 续聊时跳过既有的系统 prompt 与 initialUserMessage 注入逻辑，让 CLI 自带的会话历史保持纯净。

## 非目标

- 不自己持久化 PTY raw 输出，不复刻"视觉回放"。
- 不支持跨 CLI 续聊（从 claude 切到 codex 或反向）：切了 CLI 走新会话。
- v1 不处理"同一项目多 PTY 窗口同时续聊"的冲突；先把单会话做稳。
- 不引入第三套 AI 配置，仍复用现有主会话 AI 设置。
- 不改 CLI 自己的会话存储方式，不读 `~/.claude/projects/` 或 `~/.codex/sessions/` 的内部文件。

## 方案概览

UX 形态采用 **A2-α**：App 启动后主终端区域**不自动 spawn PTY**，先显示引导面板，两个大按钮：

```
┌─────────────────────────────────────┐
│       选择本次主会话启动方式         │
│                                     │
│   [ 新会话 ]      [ 继续上次 ]      │
│                                     │
│   当前 CLI: claude（来自 AI 设置）  │
└─────────────────────────────────────┘
```

- 点"新会话" → 走现有 spawn 路径，行为完全不变。
- 点"继续上次" → 用 `--continue` / `resume --last`，并跳过系统 prompt 注入。
- 续聊失败（CLI 立即退出且 exit code 非 0，或在 N 秒窗口内输出含 "no conversation found" 等关键字）→ 回到引导面板并附错误提示。

## 设计一：启动入口 UI

### 1. 组件位置

- 新增 `src/components/MainBootGate.tsx`：放在原本 `MainPanel` 渲染处之上。
- 状态机三态：`idle`（显示引导面板） / `running`（已 spawn，渲染原 `MainPanel`） / `failed`（显示错误提示 + 重新选择按钮）。
- App 启动 / 用户主动"重置主会话"时回到 `idle`。

### 2. 顶栏按钮的持久存在

引导面板只在 `idle` 出现一次；进入 `running` 后，顶栏新增一个**"重置主会话"**入口（与现有"模板"、"Skill 学习"按钮并列），点击：

1. 二次确认弹框 `"当前主会话将被结束，是否继续？"`
2. 确认后 kill 当前 PTY，状态回到 `idle`，再次显示引导面板。

不在 `running` 状态下额外显示"新会话 / 继续上次"按钮，避免误触。

### 3. 当前 CLI 类型可见

引导面板底部一行小字注明：**当前 CLI: claude / codex（来自主会话 AI 设置）**，并附跳转到 `AI 设置` 的链接。换 CLI 后回到引导面板需要重新点。

## 设计二：主进程改造

### 1. `cc:spawn` IPC 协议扩展

`SpawnRequest` 增加一个字段：

```ts
mode: 'new' | 'resume'
```

`ptyManager` 在收到请求时：

- `mode === 'new'`：完全沿用现有逻辑（args、系统 prompt 注入、initialUserMessage）。
- `mode === 'resume'`：
  - 把 args 改成对应 CLI 的续聊形式（见下文）；
  - **跳过** `setTimeout(... injectSystemPrompt ...)` 那一整段；
  - 仍走相同的 PTY 生命周期管理（exit 监听、resize、broadcast）。

### 2. args 转换

新增 `electron/cc/resumeArgs.ts`，纯函数：

```ts
export function buildResumeArgs(
  command: 'claude' | 'codex',
  originalArgs: string[]
): string[] {
  if (command === 'claude') {
    return ['--continue', ...filterConflictingArgs(originalArgs, ['--continue', '-c', '--resume', '-r'])]
  }
  // codex 的 resume 是子命令
  return ['resume', '--last', ...filterConflictingArgs(originalArgs, ['resume', 'fork'])]
}
```

- `filterConflictingArgs` 保证不会重复传同类参数（如用户在 AI 设置里手动加了 `-c`）。
- 注意 codex 的 `resume` 是子命令而不是 flag，要放在最前面。
- 不动 `--model` / `--effort` 等非冲突参数。

### 3. 续聊失败检测

CLI 进程在 N 秒内退出（默认 5s）且 exit code != 0，视为续聊失败。

`ptyManager` 在 `mode === 'resume'` 时：

```ts
const failGuard = setTimeout(() => { /* mark as healthy after 5s */ }, 5000)
proc.on('exit', ({ exitCode }) => {
  if (!healthy && exitCode !== 0 && mode === 'resume') {
    broadcast('cc:resume-failed', { sessionId, exitCode, tail: lastChunk })
  }
})
```

渲染进程收到 `cc:resume-failed` → 回到 `idle` 状态，并把 `tail`（最后 ~1KB 输出）作为错误详情可展开展示。

不做自动重启，由用户决定下一步。

### 4. CLI 是否登录、PTY 是否可用，沿用现有错误路径

`PtyCCProcess.start()` 抛出的异常（找不到 binary 等）路径不动，仍走 `cc:spawn` 的 `{ ok: false, error }` 返回。

## 设计三：渲染进程集成

### 1. `MainBootGate` 实现要点

```tsx
type Phase =
  | { kind: 'idle' }
  | { kind: 'spawning'; mode: 'new' | 'resume' }
  | { kind: 'running'; sessionId: string }
  | { kind: 'failed'; reason: string; tail?: string }
```

- 点"新会话"或"继续上次" → `spawning` → 调 `window.api.cc.spawn({ ..., mode })`。
- 收到 `cc:resume-failed` → `failed`。
- 收到 `cc:spawned` ack → `running`。
- 顶栏"重置主会话"点击 → kill + 回 `idle`。

### 2. 与现有 `App.tsx` 的接合

`App.tsx` 当前在挂载时直接渲染 `MainPanel`。改成：

```tsx
{bootPhase.kind === 'running'
  ? <MainPanel sessionId={bootPhase.sessionId} />
  : <MainBootGate phase={bootPhase} onChoose={...} onReset={...} />}
```

`MainPanel` 内部不动，挂载时仍按 props 中的 `sessionId` 订阅数据流。

### 3. 项目切换 / 窗口刷新

- 切换项目：当前已有"重启主会话"逻辑（kill + 新 spawn）。改造为 kill + 回 `idle`，强制用户再次选择。这样跨项目时也能选续聊。
- 窗口刷新（Cmd+R / Ctrl+R）：渲染进程状态重置，但主进程 PTY 仍在跑。已有保活机制不动；新增逻辑：刷新后 renderer 询问主进程是否已有 session，有就直接进 `running`，没有就 `idle`。

## 设计四：与 AI 设置 / 多 CLI 的关系

- AI 设置中切换主会话 CLI（claude ↔ codex）→ 触发"重置主会话"流程（如果当前在 `running`），自动 kill + 回 `idle`。
- 不在 AI 设置里增加"自动续聊"开关；A2 显式入口已经表达意图。

## 测试设计

### 1. `resumeArgs` 单元测试

- claude：基础情况返回 `['--continue']`。
- claude：原始 args 含 `-c` 或 `--continue` → 不重复。
- claude：原始 args 含 `--resume xxx` → 续聊覆盖为 `--continue`，移除 `--resume xxx`。
- codex：基础情况返回 `['resume', '--last']`。
- codex：原始 args 含 `resume` 子命令 → 不重复堆叠。
- 非冲突参数（`--model`、`--effort` 等）原样保留。

### 2. `ptyManager` 集成测试（mock PtyCCProcess）

- `mode: 'new'` → 调用 `injectSystemPrompt`。
- `mode: 'resume'` → 不调用 `injectSystemPrompt`。
- `mode: 'resume'` 且 proc 在 5s 内 exit != 0 → 广播 `cc:resume-failed`。
- `mode: 'resume'` 且 proc 5s 内未退出 → 不广播失败，视为成功。

### 3. `MainBootGate` 组件测试

- 初始 `idle`，渲染两个按钮。
- 点"新会话" → 进入 `spawning`，调 spawn 传 `mode: 'new'`。
- 点"继续上次" → 进入 `spawning`，调 spawn 传 `mode: 'resume'`。
- 收到 `cc:resume-failed` → 回到 `failed`，显示 tail。
- 在 `failed` 点"重新选择" → 回 `idle`。

### 4. 端到端手工验证

- 在 claude 项目里完整聊几轮 → 关 app → 重启 → 点"继续上次" → 终端应出现 claude 自己的 resume 回放。
- 同上但选 codex → 验 codex `resume --last` 行为。
- 在一个全新项目第一次点"继续上次" → 应出现 `failed` 状态并显示 CLI 的错误提示。
- AI 设置里切 CLI → 应触发"重置主会话"流程。

## 风险与取舍

### 1. CLI 续聊行为可能不完全直观

`claude --continue` 会回放最近一次会话，但具体渲染（是否打印历史消息、是否走"摘要展示"）取决于 CLI 版本。用户可能期望"看起来跟关 app 之前完全一样"，但实际只是"对话上下文续上、视觉布局由 CLI 自己决定"。在引导面板上小字注明"续聊将由 CLI 自身回放历史，显示可能与上次不完全一致"即可。

### 2. 多 PTY 同项目并发

v1 不做冲突检测，两个同项目窗口都点"继续上次"会续到同一份历史，CLI 内部锁机制各异，可能产生竞争。文档上写明"建议每个项目同时只开一个主会话"。

### 3. CLI 版本兼容

`claude --continue` 与 `codex resume --last` 是当前版本可用的稳定选项。后续若 CLI 改名或行为变化，`resumeArgs.ts` 是唯一改动点，便于回归。

### 4. 续聊不等于"项目元数据更新"

续聊跳过了 `buildSystemPrompt`，因此项目重命名、方案路径变更等元数据更新 CLI 不会感知。v1 接受这个限制；用户若需要让 CLI 重新读最新方案，应主动"新会话"。引导面板小字提示这一点。

### 5. 与 FORCE_COLOR 修复的关系

主题一的 `FORCE_COLOR=3` 注入在 `PtyCCProcess.ts` 内统一处理，所有 `mode` 都生效，不需要为本主题特别处理。

## 最终结论

主会话改为显式"新会话 / 继续上次"二选一入口，复用 Claude Code 与 Codex 原生续聊参数（`--continue` / `resume --last`），不自建 PTY 持久化层。续聊时跳过系统 prompt 注入，让 CLI 自带的对话历史保持纯净；续聊失败明确报错回到入口，由用户决定下一步。所有改动集中在 `resumeArgs.ts`、`ptyManager.ts`、`MainBootGate.tsx` 三处主要文件，对现有"新会话"路径零侵入。
