Codex TUI 状态显示与内部模型交互

Codex 的状态不是由一个单一 enum 控制，而是由 Turn 生命周期、模型流式事件、工具执行事件和 TUI transcript cell 共同驱动。 因此 Working、Ran、Explored、Thought 这些展示状态不能混为一个任务完成状态。

核心结论

Turn 级状态

Working 表示当前用户轮次还在运行。真正的完成边界是 TurnCompleted，不是某条消息或某条命令显示完成。

Cell 级状态

Running、Ran、Exploring、Explored 是 transcript 里的工具/命令展示状态，只代表某个动作的生命周期。

模型事件

Thought / reasoning 是模型推理摘要或 TUI 渲染出来的思考块，不应该当成最终回复转发到 IM。

最重要的边界 IM 转发不能抓 TUI 可见文本。终端可见文本里会混入用户输入、状态栏、推理摘要、工具执行记录、prompt echo 和最终回复。稳定方案必须只转发协议层最终内容，或者只转发当前 replyId 标记内的内容。

顺序与保证

  • Turn 级：TurnStarted 最先、TurnComplete 最后（携带 last_agent_message）。
  • item 级：每个 item 必然 Started → Delta×N → Completed；所有 Completed 都在 TurnComplete 之前。
  • 正文：Delta 喂 TUI 打字机；Completed 喂 IM bridge（整段）。
  • IM：每 replyId 只发一次，发的是带闭合 marker 的最终答复，通常紧邻 TurnComplete 之前。
  • steer：运行中新来的输入在 run_turn 下一圈开头被吸进当前 Turn（详见“任务插入与 steer”）。

Turn 时序

Turn 可以理解成“用户发起的一轮任务”。用户输入一条消息后，Core 会创建一个 Turn， 这个 Turn 负责组织上下文、调用模型、处理模型请求的工具调用，并在模型最终停止后给 TUI 一个完成通知。 Working 是 Turn 级状态，它从 TurnStarted 开始，到 TurnCompleted 清理。

阶段说明

阶段 内部动作 TUI 显示 关键判断
1. 用户输入 用户在 TUI 输入任务，TUI 把输入提交给 Core / Session。 输入框内容进入 transcript。 此时还没有开始真正执行 Turn。
2. Turn 启动 Core 创建当前 Turn，并向 TUI 发出 TurnStarted。 进入 Working，状态栏和标题进入运行态。 这是“任务开始”的边界。
3. 模型请求 Turn 把用户输入、系统约束、历史消息、上一轮工具结果组装后发送给模型。 如果模型输出 reasoning，TUI 可能显示 Thought。 Thought 不是最终回复，只是推理展示。
4. 模型流式输出 模型可能输出正文，也可能请求工具调用；如果请求工具，Turn 会暂停等待工具结果。 正文会流式显示；工具会进入 Running 或 Exploring。 正文出现不代表任务完成，模型后面仍可能继续调用工具。
5. 多轮循环 工具结果写回 history 后，Turn 再次请求模型，直到模型不再请求工具并给出最终停止信号。 可能反复出现 Ran、Explored、Thought、正文。 Ran / Explored 只代表单个工具动作完成。
6. Turn 完成 Turn 返回 Completed、Failed 或 Interrupted 给 Core，Core 再通知 TUI。 TUI 清理 Working，必要时启动队列里的下一条输入。 TurnCompleted 才是这一轮任务的结束边界。

因此看 TUI 时要区分两件事：Working 表示这一轮 Turn 还活着； Thought、正文、Running、Ran、Explored 都只是这个 Turn 内部过程中的展示片段。

什么时候会调用大模型

一轮 Turn 里不一定只调用一次大模型。Codex 的核心循环是“模型请求 -> 解析模型输出 -> 如果有工具调用就执行工具 -> 把工具结果写回上下文 -> 再次请求模型”。 因此大模型调用点主要有三个：

  • 首次调用：用户输入进入 Turn 后，Core 组装系统提示、开发者约束、历史消息、当前用户消息，然后发起第一次模型请求。
  • 工具后续调用：模型要求执行命令、读文件、搜索、改文件等工具时，Turn 会先执行工具；工具完成后，工具结果会作为新的上下文追加回 history，然后再次调用模型。
  • 中途 steer 调用：如果当前 Turn 期间用户又输入了 B，默认会作为 steer 立即注入正在跑的 Turn，在下一个步骤边界随该 Turn 的下一次模型请求一起送出（详见“任务插入与 steer”一节）；只有 Plan 流式 / 被拒 steer 等特例才排队、结束后另起新 Turn。

模型每次返回的内容会被分类处理：reasoning 会进入 Thought 展示；assistant 文本会进入正文展示； tool call 会进入工具执行流程；如果模型不再请求工具并给出最终停止信号，这一轮 Turn 才会进入完成阶段。

新模型同步策略

“服务端发布新模型”和“本地编译的 Codex 需要同步上游源码”不是一回事。 Codex 本身有模型目录发现和缓存机制，会通过远端 /v1/models 获取模型列表，并把结果缓存在 models_cache.json。因此，如果服务端只是新增了一个模型 slug 或更新了普通模型元数据，本地 Codex 通常不需要立刻同步上游源码；它刷新模型目录后就应该能看到或使用新模型。

需要同步上游源码的情况，主要是新模型带来了客户端必须理解的新协议、新能力或新的默认策略。 例如返回事件结构变了、工具调用协议变了、reasoning 字段语义变了、模型能力标记本地无法解析，或者上游 Codex 为这个模型专门改了默认模型选择、模型迁移提示、service tier、上下文窗口等客户端逻辑。

判断标准

变化类型 是否需要同步上游 Codex 原因
服务端只新增模型 ID，例如新模型出现在 /v1/models。 通常不需要。 Codex 的模型管理层会刷新远端模型目录，并缓存到 models_cache.json。
模型可用，但本地 UI 没有展示最新默认名称、排序或描述。 不一定需要。 如果只是展示问题，可以先看模型目录缓存和本地配置；不必为了文案或排序立即升级二进制。
新模型要求新的请求字段、响应事件、工具调用格式或 reasoning 解析方式。 需要。 这些是客户端协议兼容性问题，旧二进制可能无法正确解析模型输出。
上游 Codex 针对新模型改了默认模型、模型迁移、上下文窗口、service tier 或能力判断。 需要评估，通常建议同步。 这些逻辑会影响启动默认值、模型选择器和实际请求参数。
只是服务端灰度、限流、可用区域或账号权限变化。 通常不需要。 这是服务端或账号侧问题，同步源码不一定能解决。

建议流程

新模型发布后，先用当前本地 Codex 做兼容性验证：刷新模型目录，选择新模型，发一个普通问题，再跑一个最小工具调用任务。 如果这两类任务都能正常完成，并且 IM bridge 仍然只回传 remote-im-reply 内的最终内容，就不必马上同步上游。

如果验证失败，再看失败类型。模型不出现在列表里，优先检查 /v1/models、认证和缓存； 模型能选但回复解析异常、工具调用异常、TUI 状态错乱或 IM hook 收不到 assistant message，则应同步上游 Codex， 然后把 Multi-AI Code 的 IM patch 重新 rebase 上去。

本项目的同步原则 本地维护 Codex fork 之后，不能把“模型发布”当成“必须升级源码”的硬规则。更稳的策略是： 模型目录变化靠 /v1/models 自动发现；协议和客户端能力变化才同步上游；同步后必须跑 Codex 启动、模型选择、工具调用和 IM 回传回归。

全局安装版和自编译版为什么可能不一致

本机可能同时存在多个 Codex。全局命令 codex 可能来自 npm 安装路径，例如 /Users/hongqingwan/.real/.bin/node/bin/codex；Multi-AI Code 启动的可能是 third_party/aicli/codex/codex-rs/target/debug/codex 或 release 产物。 这两个二进制即使名字一样，也可能来自不同源码版本。

如果全局安装版拿不到新模型，而自编译版可以拿到，优先看三个差异：第一，版本是否不同； 第二，是否读的是同一个 CODEX_HOME 和 models_cache.json； 第三，全局版本是否太旧，缺少新的模型目录刷新、模型能力解析或服务端 X-Models-Etag 处理逻辑。

即使账号完全相同，两个 Codex 也可能拿到不同模型列表，因为 Codex 拉取模型目录时会把 client_version 传给服务端。当前源码中 fetch_and_update_models() 会调用 client_version_to_whole()，然后执行 endpoint_client.list_models(&client_version)。 模型缓存同样会校验 client_version，版本不匹配时不会复用旧缓存。

检查项 命令 用途
确认实际二进制 which -a codex 判断当前 shell 里的 codex 到底来自全局 npm，还是项目内置路径。
确认版本 codex --version 服务端可能根据这个客户端版本返回不同模型目录。
比较远端模型目录 codex debug models 直接看当前二进制实际拿到的服务端模型列表。
比较内置模型目录 codex debug models --bundled 排除二进制自带模型目录是否不同。

当前仓库的集成目标应该是固定使用内置路径下的 Codex，并在 UI 或日志里明确展示实际启动路径和版本。 这样用户机器上即使另外安装了一个 codex，也不会影响 Multi-AI Code 内部 AICLI 的模型列表和 IM 回传行为。

状态说明

状态 含义 进入条件 离开条件
Idle 没有正在执行的 Turn。TUI 不应该显示 Working。 程序启动后无任务，或上一轮 Turn 已结束。 用户输入新任务，进入 Starting。
Starting 用户输入已经提交，Core 正在创建当前 Turn。 TUI 把用户消息提交给 Core / Session。 Core 发出 TurnStarted 后进入 Working。
Working 当前 Turn 正在运行，是 TUI 顶部/标题的任务级状态。 收到 TurnStarted。 开始模型请求后进入 ModelStreaming；整个 Turn 完成后最终回到 Idle。
ModelStreaming Codex 正在等待或处理模型的流式返回。 Turn 组装上下文后发起模型调用，或者工具结果写回 history 后再次调用模型。 根据模型返回内容进入 reasoning、assistant 文本、工具执行，或者完成阶段。
ReasoningVisible 模型返回 reasoning / thought 类事件，TUI 显示 Thought。 模型流中出现 reasoning 事件。 reasoning 片段处理完后回到 ModelStreaming，继续等待模型后续输出。
AssistantStreaming 模型返回 assistant 正文，TUI 流式显示文本。 模型流中出现 assistant text。 文本片段结束后回到 ModelStreaming；如果没有工具调用且模型停止，则进入 Completing。
ToolRunning 模型请求执行工具，Turn 暂停下一次模型调用，先等待工具结果。 模型返回 tool call / command request。 根据工具类型进入 Exploring 或 Running。
Exploring / Explored 探索类工具状态。读文件、列目录、搜索时显示 Exploring，完成后显示 Explored。 工具被识别为 read / list / search。 工具结果写回 history 后回到 ModelStreaming，触发下一次模型调用。
Running / Ran 普通命令状态。build、test、git、脚本运行时显示 Running，完成后显示 Ran。 工具不是纯探索类命令。 命令结果写回 history 后回到 ModelStreaming，触发下一次模型调用。
Completing 模型没有继续请求工具，当前 Turn 准备收尾。 模型给出最终停止信号，或者当前流程进入错误/中断收尾。 Core 发出 TurnCompleted(Completed/Failed/Interrupted)。
Completed / Failed / Interrupted Turn 的终态。分别表示正常完成、失败、被中断。 Core 通知 TUI 当前 Turn 完成。 TUI 清理 Working，必要时启动 pending 队列里的下一条输入，然后回到 Idle 或进入下一轮 Starting。
判断任务是否完成 看 TurnCompleted，不要看 Ran、Explored 或正文是否已经出现。 后三者都只是当前 Turn 内部的局部展示状态。

把同一任务改成走 IM 路径——在 prompt 末尾按 buildRemoteImAicliPrompt 附上 reply marker 指令 （replyId = rim-demo0001），再跑一次 codex exec --json。这轮出现 5 段 agent_message，逐条看谁带 marker：

第 23 条正文原文（含 marker，位于 turn.completed 前一条）：

<remote-im-reply id="rim-demo0001">

</remote-im-reply id="rim-demo0001">

把这 5 段正文按主仓 buffer 方式拼接，跑真实的 extractRemoteImReplyOutput(buffer, { replyId: 'rim-demo0001' })，结论：

  • 只提取到第 23 条 marker 内的内容（4 条前导旁白没有 marker，被全部忽略），标签被剥掉，得到干净的约 230 字回复——这才是发回 IM 的内容。
  • 发送点：marker 落在最终答复那条 agent_message 的 item.completed（第 23 条）上，紧邻 turn.completed（第 24 条）之前——不是 TurnCompleted 触发。
  • 每 replyId 只发一次：forwardedReplyId 去重，后续再 flush 也不会重复发。
  • 提取是逐行匹配的：marker 标签必须独占一行（模型按指令照做了）。实测若把多段 agent_message 无换行直接拼接，open 标签失去独立行会导致提取为空。生产链路（ipc.ts 的 state.buffer += text）正是无分隔拼接——实际靠“中间 flush 会把不含 marker 的旁白 buffer 清空”兜底：旁白先到、flush 清掉，最终答复再来时 open 标签就在行首。若旁白与最终答复落在同一个防抖窗口内被拼在一起，提取会失败，防御性改法是监听器拼接时补一个换行。
结论（marker 版） Codex 对每段正文都发 bridge（本例 5 段），但只有带闭合 marker 的最终答复（第 23 条）被 extractRemoteImReplyOutput 提取并发回 IM，且每 replyId 仅一次。 本次 phase 全是 None 无法区分——marker 才是判定“最终回复”的唯一可靠依据。

任务 A 过程中插入任务 B

当任务 A 还在执行时用户又输入 B，codex 默认走的是 steer（中途注入），而不是“排队等下一轮”。 TUI 判定当前 agent turn 正在运行，就把 B 立即提交给 Core（不进队列）；Core 把 B 注入正在跑的 Turn A 的 pending input。它不会打断已经发出的那一次模型请求，也不会打断正在执行的工具，而是在当前步骤结束、 下一次调用模型之前把 B 取出来一起送进去。也就是说，B 被折进同一个 Turn A 继续执行，通常不会新开一个 Turn B。

只有少数特例才退回“排队 → 结束后开新 Turn”：例如 Plan 流式输出中（is_plan_streaming_in_tui()）、Plan 模式， 或 B 想 steer 一个不可 steer 的轮次（review / compact）被拒后进 rejected_steers_queue 重试。 这些情况才由 maybe_send_next_queued_input() 在 turn 空闲后送出，成为新的 Turn。

阶段说明

阶段 内部动作 对模型调用的影响 TUI 表现
1. A 正在执行 Turn A 已在运行，正处于某个“模型请求 → 工具执行”的步骤中。 已经发出的这一步不受影响，照常跑完。 显示 Working，可能出现正文、Thought 或工具状态。
2. 用户输入 B（当作 steer） TUI 见 agent_turn_running == true，把 B 当 steer 立即提交给 Core（submit_op），不进 pending 队列。 不改写正在飞行中的那次模型请求。 B 显示为本轮的 pending steer。
3. Core 注入当前 Turn inject_if_running() 把 B 追加进 Turn A 的 pending input。 B 暂存于 Turn A，等下一个步骤边界。 仍是同一个 Turn A，不新开 Turn。
4. 当前步骤结束，B 生效 run_turn 循环在下一步开头 get_pending_input() 取走 B。 下一次调用模型时带上“A 的工具结果 + B”，A 与 B 合流。 继续 Working，仍属于 Turn A。
5. 整轮收尾 模型不再请求工具、给出最终停止信号，Turn A 结束。 A 与 B 在同一轮里一起完成。 TurnCompleted(A)，清理 Working。

核心目标是：既不打断已经发出的那一步（模型请求 / 工具执行），又能让 B 尽快生效——于是选在步骤与步骤之间 把 B 折进当前 Turn。所以 B 是“注入当前这一轮的下一步”，不是“排队等下一轮”，更不是“强行改写正在飞行的请求”。 真正打断 A 是另一条中断流程，不等同于这里的 steer。

IM 消息钩子接入点

Remote IM 的链路分成两个方向：一是“IM 消息进入 AICLI”，二是“AICLI 回复回传 IM”。 这份文档只讨论 Codex：Codex 因为已经接入源码，所以不再只依赖终端可见文本，而是通过隐藏启动参数接入结构化输出桥。 Claude 仍然走 transcript 读取路径，这部分不要和 Codex 混在一起改。

这个 hook 嵌入在两个流程里：第一个是“Codex 会话启动流程”，主仓在启动 Codex PTY 时把 bridge endpoint 通过隐藏参数传进去； 第二个是“assistant message 完成流程”，Codex 在 assistant message item 完成时把正文发回主仓 bridge。 也就是说，hook 不是单独跑的后台任务，而是分别挂在 Codex 启动和 Codex assistant 输出完成这两个关键点上。

IM 回复到底在哪个阶段发出（对照状态区）

一句话结论 IM 回复既不在 Working / Thought / Ran / Explored 这些状态区元素上发出， 也不严格在 TurnCompleted 发出，而是在“Codex 写完一段 assistant 正文（AgentMessage）”这个点上； 并且只有那段正文里带着闭合的 <remote-im-reply id="…"> marker 时，主仓才真正把它发给 IM。

两层门控，缺一不可：

  • Codex 侧（发给 bridge）：每完成一段 assistant 正文（on_agent_message_item_completed）就无条件 send_assistant_text()。 它对 MessagePhase::Commentary（工具调用之间的前导/旁白）和 MessagePhase::FinalAnswer（最终答案）都发， 所以一轮里可能发好几次，而不只发最终答案。reasoning / Thought 不是 AgentMessage，不会进 bridge。
  • 主仓侧（真正发给 IM）：所有 assistant 正文先累积进 buffer，只有出现当前 replyId 的闭合 marker 时 extractRemoteImReplyOutput() 才返回内容，经防抖 flush 后发出；forwardedReplyId 保证每个 replyId 只发一次。

注入提示（buildRemoteImAicliPrompt）要求 Codex 只把最终回复包进 marker、marker 外文本忽略。 所以实际效果是：Commentary 旁白虽然也发到了 bridge，但没有 marker，累积后被忽略；只有“最终答案”那段正文（带闭合 marker）经 flush 后才发一次。 这段正文完成通常在 TurnCompleted 之前一点，所以“等 TurnCompleted 才发”并不准确。

状态区元素 Codex 是否发 bridge 是否触发 IM 发送 说明
Working / 状态栏 否 否 轮次级运行态，不是 assistant 正文。
Thought / reasoning 否 否 推理摘要不是 AgentMessage，不进 bridge。
Running/Ran/Exploring/Explored 否 否 工具/命令展示，不是 assistant 正文。
assistant 正文 · Commentary（旁白/前导） 是 否 发到 bridge 但没有 marker，累积后被忽略。
assistant 正文 · FinalAnswer（最终答案，带 marker） 是 是（每 replyId 一次） buffer 出现闭合 marker → flush → 发 IM。
TurnCompleted — 否（不是触发点） 只是轮次结束边界；发送已由 marker 触发，通常更早。

Remote IM 输出转发流程 addAicliStructuredOutputListener 追加 buffer，flushRemoteImOutputSession() 提取 marker。 只把当前 replyId marker 内的内容发给 IM。 避免旧输出、prompt echo、状态噪音被误发。

1. 主仓启动 AICLI 时注入结构化输出桥

Electron 的 PTY 管理层会判断当前启动的是哪个 AICLI。对于 codex， 主仓会创建结构化输出桥；claude 不走这条链路。

  • electron/cc/ptyManager.ts 里的 structuredOutputProvider(command) 会识别 codex。
  • 如果命中 codex，会调用 createAicliStructuredOutputBridge(sessionId, 'codex')。
  • 这个 bridge 会在本机开一个 127.0.0.1 TCP server，生成带 token 的 endpoint。
  • PTY 启动参数追加隐藏参数：--multi-ai-code-im-ipc tcp://127.0.0.1:port?token=xxx。

这一步的作用是给 Codex 一个“专用回传通道”。Codex TUI 仍然正常显示，IM 回传则通过 TCP bridge 进入主仓，不再从终端屏幕文本里硬抓。 

flowchart TD
    A[用户启动 Codex 会话] --> B[ptyManager 判断命令是 codex]
    B --> C[主仓创建 StructuredOutputBridge]
    C --> D[本机监听 127.0.0.1 TCP 端口]
    D --> E[生成 endpoint + token]
    E --> F[启动参数追加 --multi-ai-code-im-ipc]
    F --> G[Codex 进程启动]
    G --> H[tui/src/lib.rs 初始化 bridge]
    H --> I[Codex 保存 endpoint，等待 assistant 输出完成]

2. IM 消息进入 AICLI 的顺序

手机端收到消息后，Electron IPC 会进入 remote-im:deliver-incoming-text / remote-im:deliver-incoming-audio / remote-im:deliver-incoming-image。 Router 会把消息包装成 AICLI prompt，并调用 sendUserMessageToSession() 注入当前 PTY 会话。

这里的顺序很重要：必须先把用户消息发给 AICLI，等 sendUserMessageToSession() 返回成功后， 再调用 startOutputForwarding() 建立本轮输出监听状态。这样可以避免把历史输出、启动噪音或上一轮残留误算成本轮回复。

3. Codex 源码里的钩子位置

Codex 子仓新增的是一个隐藏参数加一个 TUI 侧 bridge 模块：

  • tui/src/cli.rs 定义隐藏参数 --multi-ai-code-im-ipc。
  • tui/src/lib.rs 在 TUI 启动时调用 multi_ai_code_im_bridge::init(multi_ai_code_im_ipc) 保存 endpoint。
  • tui/src/multi_ai_code_im_bridge.rs 解析 endpoint，校验 token，并通过 TCP 发送 JSON line。
  • tui/src/chatwidget/streaming.rs 在 on_agent_message_item_completed() 里对每一段完成的 assistant 正文（含 Commentary 旁白与 FinalAnswer 最终答案）都调用 send_assistant_text()。

所以 Codex 的 IM hook 不是挂在 Ran、Thought 或终端输出上，而是挂在 assistant message item 完成的位置。 这比抓 TUI 屏幕文本更稳，因为它避开了大部分状态栏、工具日志、reasoning 和 prompt echo。

什么是 assistant message item

codex 把一次会话（thread / turn）表示成一串 item（ThreadItem 枚举， app-server-protocol/src/protocol/v2/item.rs）。每种 item 是时间线上的一类条目，彼此并列： UserMessage（用户输入）、AgentMessage（assistant 正文/回复）、 Reasoning（推理，对应 Thought）、CommandExecution（命令）、FileChange（patch）、 McpToolCall / WebSearch / Plan 等。

“assistant message item” 就是其中的 AgentMessage（TUI 侧类型名 AgentMessageItem）—— 专门承载“模型说给用户看的自然语言正文”，与 reasoning、工具调用、命令、文件改动是不同的 item。结构（protocol/src/items.rs）： 

pub struct AgentMessageItem {
    pub id: String,                        // 这段正文块的唯一 id
    pub content: Vec<AgentMessageContent>, // 内容，目前只有 Text { text }
    pub phase: Option<MessagePhase>,       // Commentary | FinalAnswer | None
    pub memory_citation: Option<MemoryCitation>,
}
  • phase 区分正文类型：Commentary（轮次中途的旁白/前导，后面还会有工具或输出）、 FinalAnswer（本轮最终答复）、None（老模型不发 phase，按 legacy 语义，通常只在轮末给一段）。 源码注释原意：供 TUI 区分“中途旁白 vs 最终答案”，避免状态指示器抖动。
  • 流式两段式：先 AgentMessageDelta（增量 token，边下边显），最后 ItemCompleted{AgentMessage}（这段正文定稿）。
  • 作用：它是“纯正文”的载体（不含 reasoning / 工具日志 / 状态栏），所以 IM 桥把 hook 挂在它的完成点上，比抓终端文本稳； 带闭合 marker 的那段（通常 FinalAnswer）就是要发回 IM 的内容。

它挂在哪个状态：事件链与状态区对应

on_agent_message_item_completed() 不是自己触发的，而是 Codex core 发来 ItemCompleted 通知、 且 item 类型是 AgentMessage 时分发进来的——也就是 display-map 里的“正文”层完成， 与 Thought / Ran / Edited 等状态层是不同的 item / 通知，干净隔开： 

core: ServerNotification::ItemCompleted { item: AgentMessage { phase, .. } }
  → protocol.rs   handle_item_completed_notification()
  → replay.rs     handle_thread_item()            // 实时与 replay 共用同一分发
  → replay.rs     match ThreadItem::AgentMessage
  → streaming.rs  on_agent_message_item_completed(item, from_replay)
  → streaming.rs  send_assistant_text()           // from_replay=false 才发 bridge
状态区元素（display-map） 触发它的 core 事件 / item TUI 分发函数
正文（← IM hook 在这里） ItemCompleted { AgentMessage, phase } on_agent_message_item_completed() → send_assistant_text()
Thought / reasoning ReasoningSummaryTextDelta on_agent_reasoning_delta()
Running / Ran CommandExecution item（started / completed） on_command_execution_started() 等
Edited FileChange（patch）item on_patch_apply_begin() 等
Exploring / Explored / 工具 WebSearch / McpToolCall item on_web_search_begin() / on_mcp_tool_call_started()

AgentMessage 的 phase 再细分正文类型：Commentary（工具间前导/旁白）、 FinalAnswer（最终答案）、None（老模型只在轮末给一段）。所以它在一轮 Turn 内可能触发多次， 且都发生在 TurnCompleted（另一个 TurnStatus::Completed 通知）之前。

不是等到 TurnCompleted 才发送 Codex 侧触发点是 on_agent_message_item_completed()：每完成一段 assistant 正文就发一次（含 Commentary 旁白），并不是只发最终答案，也不是 TurnCompleted 事件本身。 主仓把这些正文累积进 buffer，只有出现当前 replyId 的闭合 marker 时才经 flush 真正发回 IM；forwardedReplyId 保证每个 replyId 只发一次。 由于注入提示只让 Codex 把最终回复包进 marker，实际就是“带 marker 的最终答案那段完成后”发出——通常略早于 TurnCompleted。

4. 主仓收到结构化输出后的转发

Codex 发回的 JSON line 不会直接发给 IM。主仓 electron/aicli/structuredOutputBridge.ts 会先校验 token，然后触发 addAicliStructuredOutputListener 注册的监听器。

electron/remote-im/ipc.ts 中的监听器把结构化文本追加到当前 session buffer，再按配置延迟 flush。 最终仍然走 flushRemoteImOutputSession()：

  • 通过 extractRemoteImReplyOutput(buffer, { replyId }) 只提取当前 replyId marker 内的内容。
  • 通过 sanitizeRemoteImAicliOutput() 做来源相关的兜底清洗。
  • 通过 createOutputChunks() 按单条长度分段。
  • 通过 broadcastOutgoingText() 发回 IM。
  • 通过 forwardedReplyId 防止同一个 replyId 被重复转发。

因此完整判断是：assistant message item completed 只是“有一段候选回复可以进入 IM bridge”； extractRemoteImReplyOutput() 成功提取当前 replyId 内的闭合内容，才是“可以发 IM”的条件。 TurnCompleted 只是整轮任务生命周期的结束边界，不是当前 IM 转发链路的唯一发送触发点。

IM 转发边界

Remote IM 的转发逻辑必须绕开 TUI 展示层。原因是 TUI 可见区域天然包含大量非回复内容：

  • 用户输入和 prompt echo。
  • Working、状态栏、模型名、上下文额度等状态信息。
  • Thought / reasoning 摘要。
  • Running、Ran、Exploring、Explored 等工具执行展示。
  • 命令输出、错误日志、测试日志。

因此 IM 只应该认当前请求的 reply marker： 

<remote-im-reply id="rim-当前请求">
需要发回 IM 的最终内容
</remote-im-reply id="rim-当前请求">

marker 外的任何内容，包括 reasoning、工具日志、状态栏、历史残留，都应该被视为噪音。
