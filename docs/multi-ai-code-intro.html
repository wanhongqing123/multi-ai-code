EXECUTIVE PRODUCT SHOWCASE

Multi-AI Code

把方案阅读、AI 编码推进、Diff 审查和批注回灌放进一个本地可控的工作台里，让 AI 真正适合真实项目推进。

为什么需要这个产品

不是 AI 不够强，而是 协作流程不够顺

在真实项目里，最耗时的不是写第一版代码，而是方案确认、上下文保持、审查反馈和继续修改这几个来回。

方案散落

方案如果只在聊天记录里，过两天就很难复用，也很难交接。

长会话偏移

目标容易被最新一次修改带偏，最后变成“修着修着忘了原意”。

审查与执行断开

Diff 看完后要手工复述给 AI，容易丢行号、丢上下文、丢重点。

工具切换成本高

Claude Code 和 Codex 可以都用，但最好有统一的入口和流程。

产品定位

一个面向本地仓库的 AI 编码工作台

它不是普通聊天壳，也不是一次性的脚本，而是把方案、执行、审查和反馈都放进同一条工作流。

它做什么

  • 把方案变成显式文件，而不是隐藏在聊天记录里
  • 把代码实施放进一个本地 AI 会话里推进
  • 把 Diff 审查变成结构化批注，而不是口头描述
  • 把批注继续回灌给 AI，形成闭环

它的边界

  • 不替代代码编辑器，而是补足 AI 协作环节
  • 不把所有能力做成云端服务，而是优先保证本地可控
  • 不依赖单一 CLI 工具，Claude Code 和 Codex 都能接入
  • 不把技术实现当卖点，重点是工作流是否顺手

核心能力总览

四个能力，拼成一条可用的产品闭环

01

主会话终端

一个 PTY 驱动的主会话，负责方案推进、代码修改和过程输出。

02

方案导入与预览

支持内部方案和外部方案，先看清方向，再决定是否启动。

03

Diff 审查与回灌

把代码修改后的反馈变成结构化批注，直接发回 AI 继续处理。

04

双 CLI 配置

Claude Code 与 Codex 可按团队习惯选择，主会话和辅助能力可分别配置。

典型使用流程

实际使用时，不是“随便聊到哪算哪”，而是按一个清晰的节奏往前推进。

STEP 1

选仓库

先确定当前工作的目标仓库，后续方案和修改都围绕它展开。

STEP 2

选方案

从已有方案中挑选，或导入外部方案，再决定要不要启动。

STEP 3

启动 AI

AI 先读方案，先总结，再等待确认；确认后才进入实施。

STEP 4

Diff 审查

改完后看差异，逐行标注哪里需要修、哪里需要补。

STEP 5

批注回灌

把批注直接发回 AI CLI，让下一轮修改接着上一次的上下文继续。

方案驱动能力

方案不是附件，而是整个流程的起点

Multi-AI Code 会先把方案变成明确的工作输入。对于外部方案，只发路径，不把整篇内容塞进会话里刷屏。

内部方案

方案已经在当前仓库里时，先让 AI 读内容、总结目标，再进入执行。

外部方案

如果方案来自外部文件，只把绝对路径交给 AI，让它自己读取，减少上下文浪费。

用户确认前不修改代码

先理解方案、先确认方向，避免 AI 在目标未确认时提前开改。

PLAN PREVIEW 方案预览页先把内容讲清楚，再决定是否启动

Diff 审查与批注回灌

审查不是结束，而是 下一轮修改的输入

怎么看

  • 看工作区改动、最近提交或指定 commit
  • 左右对照查看修改前后差异
  • 支持对单行或多行做批注
  • 批注可以写整体意见，也可以点到具体行上

怎么回灌

  • 批注会自动整理成结构化说明
  • 发送后成为下一轮 AI CLI 的输入
  • AI 会基于这些反馈继续改代码
  • 整条链路不需要人工再复制粘贴一次

AI CLI 配置能力

Claude Code 和 Codex 都能接，按团队习惯选

共同点

  • 主会话和辅助能力都可分别选择 CLI 工具
  • 支持 binary override、附加参数、环境变量
  • 终端输出统一由本地 PTY 托管，保持交互一致

差异点

  • Claude Code：适合已有 Claude 工作流的团队
  • Codex：适合更偏工具链、命令式的使用习惯
  • 界面只暴露“选择哪个 CLI”和“怎么配”，不用关心底层协议

当前产品价值

它的价值不在“能不能写”，而在能不能把过程管住

01

过程更可控

方案先行、确认后执行，减少 AI 在目标不明确时跑偏。

02

反馈更闭环

Diff 审查的批注直接进入下一轮输入，减少人工转述。

03

切换更顺手

Claude Code 和 Codex 都能接入，不强迫团队改变既有习惯。

04

资产可沉淀

方案、批注和历史都保留在本地，后续可继续扩展成团队知识资产。

下一步机会

当前已经能展示和使用，下一步重点是更稳、更顺、更能沉淀

这部分不是“功能堆更多”，而是把当前已经有的协作闭环做得更像一个团队级产品。

更好的可读性

继续提升 markdown、长文本和代码块的展示，让内容更适合汇报和复盘。

更完整的记忆

把方案、批注、项目上下文沉淀得更系统，让下次打开同仓库时更快接上。

更强的团队使用方式

把演示、复盘和后续协作做得更标准化，方便团队持续使用，而不只是单次 demo。

结论

Multi-AI Code 现在已经不是概念稿，而是一套可以讲清楚、可以演示、也可以继续打磨的产品工作台。
