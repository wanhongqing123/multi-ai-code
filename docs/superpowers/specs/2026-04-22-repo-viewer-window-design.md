# Repo Viewer Window — Design Spec

**Date:** 2026-04-22

**Goal:** 在不改动现有 `Diff 审查` 逻辑的前提下，为 Multi-AI Code 新增一个独立的 `仓库查看` 原生窗口。该窗口用于浏览当前仓库的完整文件树与单文件源码，并支持对选中代码发起独立 AI 分析。分析历史以压缩后的项目私有记忆形式保存在 `<target_repo>/.multi-ai-code/repo-memory/`，供下次打开同一仓库时快速建立上下文。

---

## 1. 用户可见行为

- 主窗口保留现有 `Diff 审查` 按钮、弹窗和批注回灌流程，不做语义变化
- 主窗口新增一个 `仓库查看` 入口
- 点击 `仓库查看` 时，Electron 新建一个拥有系统标题栏的独立窗口
- 新窗口左侧是 VS Code 风格的文件树，右侧是当前文件完整源码
- 文件树默认过滤 `.git`、`node_modules`、`dist`、`build`、`out`
- 在源码区选中一段代码后，出现 `分析` 操作；用户输入问题后，后台启动独立 AI CLI 分析会话
- 分析结果显示在 `仓库查看` 窗口内，不打断主窗口当前编码会话
- 分析结论会压缩保存到仓库内的私有记忆目录，下次再次打开同一仓库时优先加载

## 2. 非目标

- 不替换或重写现有 `Diff 审查` 组件
- 不把 `仓库查看` 做成 modal 或主窗口内嵌 drawer
- 不在第一版里提供“应用补丁 / 修改代码 / 直接提交”能力
- 不保存完整原始长对话 transcript
- 不将仓库记忆默认提交进 Git

## 3. 顶层架构

### 3.1 主窗口

- 继续使用现有 `App.tsx` 作为项目、方案、会话和 `Diff 审查` 的主入口
- 新增 `仓库查看` 按钮和命令面板项
- 该入口只负责请求主进程创建一个独立窗口，不在主窗口内渲染 `仓库查看` UI

### 3.2 主进程

- 新增 `createRepoViewWindow()`，创建一个独立 `BrowserWindow`
- 窗口使用系统标题栏，因此天然拥有最小化、最大化、关闭
- 该窗口复用现有 renderer bundle，通过 query 参数或等价的 route 参数区分“主窗口”与“仓库查看窗口”
- 例：`index.html?window=repo-view&projectId=<id>`

### 3.3 Renderer

- 新增一个 `RepoViewerWindow` 页面组件
- `src/main.tsx` 在启动时根据 URL 参数决定挂载 `App` 还是 `RepoViewerWindow`
- `RepoViewerWindow` 自己管理文件树、文件内容、选区分析、分析历史和独立分析会话状态

## 4. 仓库查看窗口

### 4.1 窗口属性

- 标题：`仓库查看 · <项目名>`
- 默认尺寸：`1440 x 920`
- 最小尺寸：`1100 x 720`
- 系统标题栏开启，提供最小化、最大化、关闭
- 同一项目默认只允许一个 `仓库查看` 窗口；重复点击时聚焦已有窗口，而不是无限新开

### 4.2 布局

- 左侧：文件树 sidebar
- 中间：代码查看区
- 右侧：分析面板

三栏布局是第一版的推荐默认值，因为用户需要同时：

- 浏览文件树
- 阅读源码
- 查看 / 追问 AI 分析

其中中间代码区应保持最大宽度，右侧分析面板可折叠。

### 4.3 左侧文件树

- 显示当前 `target_repo` 的目录结构
- 默认过滤目录：
  - `.git`
  - `node_modules`
  - `dist`
  - `build`
  - `out`
- 按目录懒加载，而不是一次性全量扫描
- 展开 / 折叠行为独立于主窗口
- 再次打开同一项目时恢复上次展开目录和选中文件

### 4.4 中间代码查看区

- 单列只读源码视图，不使用左右 diff 布局
- 展示当前文件完整内容
- 带行号
- 右上或顶部显示当前相对路径
- 对二进制文件显示“该文件不可作为文本预览”
- 对超大文件进行保护：超过阈值先提示“文件过大，是否继续加载”

## 5. 代码分析交互

### 5.1 基本流程

1. 用户在中间代码区选中一段源码
2. 浮现 `分析` 按钮
3. 点击后打开右侧分析 composer
4. 用户输入问题，例如“这一段为什么这样设计？”、“这个函数的逻辑入口在哪里？”
5. 后台把当前文件路径、选中代码、用户问题，以及仓库私有记忆一起送入独立分析会话
6. 分析结果显示在右侧分析面板
7. 用户可继续追问，形成同一窗口内的分析线程

### 5.2 与 Diff 审查的区别

- `Diff 审查`：针对改动做审查，结果回灌给主窗口当前编码会话
- `仓库查看`：针对现有代码做逻辑分析，结果留在独立分析线程里，不直接回灌主窗口编码会话

### 5.3 会话模型

- 每个 `仓库查看` 窗口持有一条独立分析会话
- 关闭窗口时，该分析会话一并结束
- 新分析会话不显示终端，不暴露 xterm
- 分析会话默认用于解释、追踪、总结，不承载“应用修改”交互

## 6. AI CLI 配置

### 6.1 新增项目级配置

在 `project.json` 中新增一组独立配置：

```json
{
  "repo_view_ai_settings": {
    "ai_cli": "claude",
    "command": "claude",
    "args": [],
    "env": {}
  }
}
```

### 6.2 默认值

- 默认 `Claude`
- 与主窗口编码会话的 `ai_settings` 分离
- 用户可以在 `AI 设置` 中单独配置 `仓库查看分析 AI`

### 6.3 隔离原则

- 主窗口的编码 CLI 会话与 `仓库查看` 的分析会话互不影响
- 修改主窗口 AI 设置，不应隐式改掉 `仓库查看分析 AI`
- 修改 `仓库查看分析 AI`，不应影响正在运行的主窗口编码会话

## 7. 独立分析会话

### 7.1 后端管理器

- 新增 `electron/repo-view/repoAnalysisManager.ts`
- 复用现有 CLI 启动经验，例如环境清洗、路径处理、Codex/Claude binary 解析
- 该管理器维护“窗口 ID -> 分析会话”的映射

### 7.2 Prompt 约束

系统 prompt 必须明确：

- 你正在做代码逻辑分析，而不是代码实现
- 不要修改仓库文件
- 不要运行写操作命令
- 重点输出模块职责、调用链、数据流、风险点和推理依据
- 回答要引用相对路径和行号

### 7.3 会话输入

每次发起分析时注入：

- 当前仓库根路径
- 当前选中文件相对路径
- 当前选区源码
- 用户问题
- 仓库私有记忆摘要
- 可能的相关文件摘要

### 7.4 会话输出

分析结果以结构化块形式返回到窗口右侧面板：

- 摘要
- 关键路径 / 关键函数
- 相关文件
- 风险 / 注意点
- 后续建议问题

第一版只要求可读显示，不要求强结构化 JSON UI。

## 8. 仓库私有记忆

### 8.1 存储位置

- 路径：`<target_repo>/.multi-ai-code/repo-memory/`
- 该目录用于帮助 AI 快速熟悉同一仓库，不面向 Git 共享

### 8.2 Git 私有化

- 在首次创建记忆目录时，主进程要确保 `.git/info/exclude` 包含：

```gitignore
.multi-ai-code/repo-memory/
```

- 这是本地私有排除，不修改仓库 `.gitignore`

### 8.3 目录结构

- `project-summary.md`
  - 项目整体逻辑、关键模块、术语、约束
- `file-notes/<relative-path>.md`
  - 某些核心文件的持续压缩摘要
- `recent-topics.json`
  - 最近分析主题的轻量索引，如文件、问题摘要、更新时间

### 8.4 不保存的内容

- 不保存完整长对话 transcript
- 不保存模型链路推理原文
- 不保存大段重复代码原文

## 9. 记忆压缩策略

### 9.1 写入时机

- 每次分析回答完成后，立即更新 `recent-topics.json`
- 当窗口 idle、关闭窗口，或累计若干轮分析后，触发一次增量压缩

### 9.2 压缩目标

- 把“稳定事实”写入 `project-summary.md`
- 把“文件特有结论”写入对应的 `file-notes/<relative-path>.md`
- 只保留对下次分析有帮助的高密度上下文

### 9.3 压缩原则

- 面向“让下次 AI 快速熟悉项目”
- 保留模块职责、调用链、关键约束、常见坑
- 去掉寒暄、重复问答和一次性无价值噪音

## 10. IPC 设计

### 10.1 窗口管理

- `repo-view:open-window`
  - 输入：`projectId`
  - 行为：打开或聚焦该项目的 `仓库查看` 窗口

### 10.2 仓库树

- `repo-view:list-tree`
  - 输入：`root`, `dir?`
  - 输出：某目录下的子目录与文件，按目录优先排序
  - 默认应用过滤规则

### 10.3 文件读取

- `repo-view:read-file`
  - 输入：`root`, `relPath`
  - 输出：文件文本内容、大小、是否文本文件

### 10.4 会话管理

- `repo-view:analysis-start`
  - 为窗口创建或复用独立分析会话
- `repo-view:analysis-send`
  - 发送一条分析请求
- `repo-view:analysis-stop`
  - 关闭该窗口的分析会话
- `repo-view:analysis-data`
  - 流式回传分析输出
- `repo-view:analysis-status`
  - 回传 idle / running / error 等状态

### 10.5 记忆

- `repo-view:memory-load`
  - 读取项目摘要、相关文件摘要、最近主题
- `repo-view:memory-update`
  - 分析结束后更新记忆文件

## 11. 前端组件建议

- `src/repo-view/RepoViewerWindow.tsx`
  - 窗口入口页
- `src/repo-view/FileTree.tsx`
  - 左侧文件树
- `src/repo-view/CodePane.tsx`
  - 中间代码区
- `src/repo-view/AnalysisPanel.tsx`
  - 右侧分析面板
- `src/repo-view/repoViewStore.ts`
  - 当前文件、展开目录、分析状态等窗口级状态

这样可避免把 `DiffViewerDialog` 继续膨胀成双职责组件。

## 12. 错误与性能保护

- 仓库路径不存在：显示错误页，不影响主窗口
- 读取目录失败：在文件树节点上显示错误提示
- 二进制文件：右侧提示“不可预览文本文件”
- 大文件：先提示，再决定是否读取
- 文件树懒加载，避免大仓库启动时卡死
- 分析输出流式显示，避免长回答阻塞窗口

## 13. 测试

### 13.1 自动测试

- 文件树过滤规则：`.git`、`node_modules`、`dist`、`build`、`out`
- 目录懒加载返回结构
- 文本文件 / 二进制文件识别
- `.git/info/exclude` 幂等写入
- 记忆目录路径与文件名映射
- 窗口参数解析与 renderer route 分流

### 13.2 手动验收

- [ ] 主窗口点击 `仓库查看` 能打开独立原生窗口
- [ ] 再次点击同项目 `仓库查看` 会聚焦已有窗口
- [ ] 文件树能正常展开，且过滤大目录
- [ ] 点击文件后右侧显示完整当前内容
- [ ] 选中代码后可以发起分析
- [ ] 分析结果显示在右侧面板，不影响主窗口会话
- [ ] 关闭并重开同一项目 `仓库查看`，能恢复上次文件与部分记忆
- [ ] `repo-memory` 已落在仓库内，但默认不出现在 `git status`

## 14. 风险

- 独立分析会话如果完全依赖 prompt 约束，理论上仍可能尝试写操作；实现时应尽量选择只读运行策略
- 大仓库首次展开文件树可能仍有感知延迟，懒加载与过滤规则是性能核心
- 记忆压缩如果质量太差，会让后续分析产生噪音；必须限制写入内容只保留稳定事实
- 复用单一 renderer bundle 会让入口分流逻辑变复杂，但比维护第二套构建入口更稳

## 15. 决策总结

- `Diff 审查`：保持不变
- `仓库查看`：独立原生窗口
- 文件浏览：左树右码，不做双列 diff
- 代码分析：独立 AI CLI 会话
- 分析 AI：单独配置，默认 `Claude`
- 项目记忆：保存在仓库内 `.multi-ai-code/repo-memory/`
- Git 策略：通过 `.git/info/exclude` 默认本地私有，不进 Git
