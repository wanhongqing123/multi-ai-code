# Multi-AI Code 老板展示版 PPT 设计规格

**日期**：2026-04-23  
**目标产物**：一份约 10 页的产品展示型 PPT，面向老板汇报 Multi-AI Code 的产品价值与当前可用能力。  
**输出形式**：优先生成 `docs/Multi-AI-Code-介绍.pptx`，必要时同步维护生成脚本与辅助 HTML 预览稿。  

## 1. 背景与目标

当前仓库中已有介绍 PPT 与生成脚本，但内容叙事仍然偏旧版本架构，并包含已经不适合作为本次重点介绍的模块。现在需要重做一版更符合当前产品形态的介绍材料，用于老板视角的产品展示。

本次 PPT 的目标不是技术说明书，也不是纯截图拼贴，而是让老板在 10 页左右的阅读时间内快速理解：

- 这个产品解决什么问题
- 现在已经具备哪些可演示能力
- 实际使用流程是什么样的
- 为什么这个方向值得继续投入

## 2. 受众与沟通策略

### 2.1 受众

主受众为老板 / 决策者。

### 2.2 沟通策略

- 采用“产品展示”而不是“开发汇报”语气
- 内容同时覆盖“产品能力”和“实际使用流程”
- 技术实现只保留对业务判断有帮助的部分，不展开底层细节
- 控制文字密度，强调一句话结论与少量支撑点
- 让截图承担“可信度”和“落地感”，让文案承担“价值表达”

## 3. 范围与非目标

### 3.1 本次范围

纳入本次 PPT 的内容：

- 产品定位
- 核心问题与解决思路
- 主会话工作方式
- 方案导入 / 方案预览 / 方案标注
- Diff 审查与批注回灌
- Claude Code / Codex 双 CLI 配置能力
- 典型使用流程
- 当前价值与下一步空间

### 3.2 明确不纳入

本次 PPT 明确不介绍：

- `仓库查看`
- 旧版“四阶段流水线”架构作为主叙事
- 过深的 Electron / React / PTY / 数据结构实现细节
- 面向开发者的 API 级说明

## 4. 核心叙事

整份 PPT 的主线为：

1. 普通 AI 编码流程存在明显协作断点
2. Multi-AI Code 把“方案、主会话、Diff 审查、反馈回灌”串成一条闭环
3. 这个闭环已经具备可展示、可使用、可持续迭代的产品雏形
4. 它的价值不只是“能用 AI”，而是把 AI 编码过程变得更可控、更可审查、更适合真实项目推进

## 5. 页面结构

总页数控制在 10 页左右，按以下结构组织。

### Slide 1：封面

目标：建立产品第一印象。

内容：

- 标题：`Multi-AI Code`
- 副标题：面向本地仓库的 AI CLI 协作工作台
- 一句价值表达：把方案阅读、主会话推进、Diff 审查与反馈回灌放进一个本地工具里
- 主界面截图作为主视觉

### Slide 2：为什么需要这个产品

目标：先讲问题，再讲产品。

内容：

- 现有 AI 编码工具常见问题：
  - 方案和执行脱节
  - 长会话容易偏离目标
  - Diff 审查后的反馈需要手工转述
  - CLI 工具切换成本高
- 结论句：不是 AI 不够强，而是协作流程不够顺

### Slide 3：产品定位

目标：一句话说清产品是什么。

内容：

- 强调不是普通聊天壳
- 强调是面向真实仓库的本地 AI 工作台
- 核心组成：
  - 方案
  - 主会话 AI CLI
  - Diff 审查
  - 批注回灌

### Slide 4：核心能力总览

目标：给老板一个全局概念图。

内容：

- 主会话终端
- 方案导入 / 预览 / 标注
- Diff 审查与逐行反馈
- 双 AI CLI 配置

形式建议：4 卡片或 4 象限布局。

### Slide 5：典型使用流程

目标：说明真实使用路径。

内容：

- 选择仓库
- 选择或导入方案
- 启动 AI CLI
- AI 先读方案，再等待确认
- 代码实现后进入 Diff 审查
- 标注后把反馈回灌给当前会话继续修改

形式建议：横向流程图或分步编号。

### Slide 6：方案驱动能力

目标：突出这个产品和普通 AI 对话工具的不同。

内容：

- 方案文件是显式文档，而不是沉在聊天记录里
- 支持仓库内方案与外部方案
- 外部方案只传文件路径给 AI CLI，自行读取，避免长文本刷屏
- 在用户确认前，AI 先读方案，不直接修改代码

建议使用：方案预览相关截图 + 3 条价值点。

### Slide 7：Diff 审查与批注回灌

目标：这是本次展示的核心亮点页。

内容：

- 双栏 diff 审查
- 选中单行或多行做标注
- 右侧填写说明
- 一键把批注发回 AI CLI
- AI 继续基于反馈修改代码

建议使用：Diff 审查截图作为页面主视觉。

### Slide 8：AI CLI 配置能力

目标：体现产品灵活性与适配性。

内容：

- 支持 `Claude Code` 和 `Codex`
- 主会话与分析会话可独立配置，但本次不展开分析会话细节
- 支持 binary override、附加参数、环境变量
- 适应不同团队已有的 AI CLI 使用习惯

### Slide 9：当前产品价值

目标：从老板视角总结“为什么值得继续做”。

内容：

- 让方案到实现的过程更可控
- 让代码审查和 AI 修改形成闭环
- 降低团队使用 AI CLI 的组织成本
- 为后续团队协作、历史沉淀、项目级上下文管理打基础

### Slide 10：下一步机会

目标：给决策留空间。

内容：

- 更稳定的长会话体验
- 更强的 markdown / 输出可读性
- 更完善的项目级记忆与历史追踪
- 更成熟的演示与团队协作能力

结尾语气应偏“当前已具备清晰方向和可演示能力，适合继续打磨”，不夸大为完全成熟产品。

## 6. 视觉设计要求

### 6.1 总体风格

- 风格：偏产品展示
- 气质：现代、克制、有质感
- 避免：模板感过强、炫技式渐变、信息堆叠、工程文档风

### 6.2 页面语言

- 标题大、正文少
- 每页一个中心信息
- 用截图增强可信度
- 用简洁卡片与章节编号增强节奏

### 6.3 字体与排版

- 中文标题：可使用更有表现力的衬线或高质感中文字体
- 正文：使用清晰的无衬线字体
- 命令 / 文件名 / CLI 名称：使用等宽字体
- 16:9 宽屏布局

### 6.4 配色建议

- 浅色背景为主
- 深色文字为主
- 使用单一强调色点亮重点信息
- 局部可用深底封面或结束页形成层次

## 7. 素材策略

### 7.1 可复用素材

可优先使用仓库中已有截图作为基础素材，按本次叙事重新裁剪与编排：

- `docs/screenshots/主界面.png`
- `docs/screenshots/方案标注.png`
- `docs/screenshots/方案 Review.png`
- `docs/screenshots/代码 Review.png`

### 7.2 素材处理原则

- 截图不宜一页放太多
- 需要时可放大重点区域，而不是整屏缩小
- 同一页截图和文案要互相配合，不做简单拼贴

## 8. 交付物

本轮交付建议包含：

1. 更新后的 PPT 文件
2. 如有必要，同步更新 PPT 生成脚本，保证后续可重复生成
3. 如有必要，保留 HTML 预览稿便于快速审阅

## 9. 验收标准

完成后的 PPT 应满足：

- 老板不需要了解代码细节，也能看懂产品定位与价值
- 全文不再沿用旧的“四阶段流水线”作为主叙事
- 明确不介绍 `仓库查看`
- 页数控制在 10 页左右
- 风格偏产品展示，不是普通内部汇报模板
- 每页信息中心明确，不出现大段密集说明文字
- 与当前 README 和当前产品能力保持一致

## 10. 实施建议

建议优先采用“脚本生成 PPT”的方式完成本次改版，原因如下：

- 现有仓库已经有 PPT 生成脚本，可复用基础能力
- 视觉风格可以统一收敛
- 后续迭代时可重复生成，而不是手工维护二进制文件

如脚本改造成本明显高于收益，则可采用：

- 先更新 HTML 演示稿完成文案与视觉编排
- 再导出或同步生成最终 PPT

以上两种路径，优先保证最终 PPT 成品质量，而不是拘泥于单一产出方式。
