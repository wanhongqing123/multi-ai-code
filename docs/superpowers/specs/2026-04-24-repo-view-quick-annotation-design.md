# Repo View 快速批注链路精修设计

## 背景

当前 `repo-view` 窗口已经具备可用的三栏结构：

- 左侧文件树负责浏览仓库文件
- 中间代码区负责阅读源码、选区和添加批注
- 右侧批注栏负责整理当前文件的批注并发送到 AI CLI

但“选中代码 -> 添加批注 -> 在右侧整理 -> 发送”的链路仍然偏粗糙，问题主要集中在：

- 代码区虽然能选中和添加批注，但视觉反馈偏弱，缺少稳定的上下文感
- 浮动“标注”按钮像临时弹出物，存在感过强但精度不够
- 右侧批注列表更像简单数据列表，而不是“待发送批注托盘”
- 代码区与右侧卡片之间几乎没有关联反馈，编辑和整理时容易失去定位

这次设计不重构仓库查看器，也不改左侧文件树，而是在保留现有产品视觉语言的前提下，把中间代码区和右侧批注栏精修成更顺手的快速批注体验。

## 目标

- 保持当前三栏结构和整体产品风格不变，只优化代码区与右侧批注栏
- 强化快速批注链路，让用户更自然地完成“阅读、选中、标注、整理、发送”
- 让代码区保持主舞台地位，不因批注功能而变成嘈杂的审阅工具
- 让右侧栏明确承担“当前文件待发送批注托盘”的角色，而不是聊天区
- 用最小状态增强建立代码区与批注卡片之间的轻关联

## 非目标

- 不改左侧文件树结构与交互
- 不引入语法高亮系统
- 不做跨文件批注分组或全局批注中心
- 不把右侧批注栏改造成对话面板
- 不重写现有批注编辑流程，只优化入口、视觉和关联感

## 方案选择

本次在两种轻量方案之间选择：

- 方案 A：更克制的轻量方案
- 方案 B：更强调链路的轻量方案

最终选择 **方案 A**。

原因：

- 用户希望“快速批注”优先，不要打断读代码
- 用户明确要求保留现有产品风格，不希望形成明显的独立审阅模式
- 方案 A 更适合在现有代码结构上做增量优化，风险低，落地快

## 设计原则

### 1. 代码区是主舞台

中间代码区继续承担主要注意力，不用重配色或重阴影去制造“设计感”，而是通过：

- 更清晰的代码头部层次
- 更稳定的 gutter 与代码正文对比
- 更细的行背景节奏
- 更克制的空状态与选区态

来提升可读性。

### 2. 批注入口是轻动作

选中文本后仍然通过浮动按钮进入批注流程，但该按钮应表现为低干扰上下文动作：

- 不使用明显重主色块作为默认形态
- 不抢右侧发送按钮的主次
- 更像“当前选区可执行的动作芯片”

### 3. 右侧是托盘，不是聊天栏

右侧批注区负责承接当前文件的待发送批注，强调：

- 位置信息
- 摘录内容
- 批注意见

不承载长对话、不增加复杂分组、不引入重型 review 工具模式。

### 4. 关联感只做一层

代码区与右侧批注卡片之间只建立一层明确关联：

- 新增或编辑某条批注时，右侧卡片进入聚焦态
- 代码区保留对应该条批注的轻量关联感

不做多重联动动画，也不做复杂的行内工具栏系统。

## 组件设计

### CodePane

文件：`src/repo-view/CodePane.tsx`

职责保持不变：

- 显示当前选中文件内容
- 处理文本选区
- 提供添加批注入口
- 承接编辑态批注

本次改动重点：

- 代码头部从普通标题栏升级为更清晰的“文件上下文条”
- 代码行样式增加更细的层次感，但不引入语法高亮
- 文本选区高亮更稳定，用户可更清楚地意识到当前操作对象
- 浮动“标注”按钮改成更低干扰的轻按钮
- 编辑态保持在当前模式内完成，不切断用户的代码上下文

### AnalysisPanel

文件：`src/repo-view/AnalysisPanel.tsx`

职责重新明确为：

- 呈现当前文件的待发送批注列表
- 支持轻量编辑、删除、清空
- 呈现发送前摘要与可选问题输入

本次改动重点：

- 卡片样式更像“待处理批注项”，而不是简单列表条目
- 强化位置信息、摘录区和意见正文的扫读顺序
- 编辑/删除保留为轻操作，不升级成重型管理面板
- 空状态更像操作引导，而不是静态提示文案
- 底部发送区更明确表达“发送当前文件批注到 AI CLI”

### RepoViewerWindow

文件：`src/repo-view/RepoViewerWindow.tsx`

整体布局不改，只做最小状态增强。

新增状态：

- `activeAnnotationId`
  - 表示当前正在编辑或被聚焦的批注
  - 用于右侧卡片高亮和代码区轻关联
- `recentlyAddedAnnotationId`
  - 表示刚被添加到当前文件批注队列的项目
  - 用于右侧卡片短暂强调

保留现有状态模型：

- `annotations` 仍由窗口层统一持有
- 批注仍按 `filePath` 归属
- 切换文件时仅清理编辑态，不改未发送批注保留策略

## 数据流

核心数据流保持当前结构：

1. 用户在 `CodePane` 中选中文本
2. `CodePane` 生成 `RepoSelection`
3. 用户通过浮动按钮进入批注编辑
4. `CodePane` 通过 `onAnnotateSelection(...)` 将批注提交给 `RepoViewerWindow`
5. `RepoViewerWindow` 更新当前 `annotations`
6. `AnalysisPanel` 基于当前文件的批注数组渲染待发送卡片
7. 用户在右侧点击发送
8. `RepoViewerWindow` 调用既有 `buildCliInjectionText(...)` 和 `analysisSend(...)`

只新增两个轻状态分支：

- 新增批注时设置 `recentlyAddedAnnotationId`
- 点击编辑或选中特定卡片时设置 `activeAnnotationId`

## 交互细节

### 代码区

- 文件头部显示文件路径主信息与基础元信息（如行数、字节数）
- 代码区空状态更明确地提示“先从左侧选择文件”
- 读取中文件时维持简单、安静的加载提示
- 选中文本后出现轻量浮动“标注”按钮
- 点击浮动按钮后进入现有批注编辑流程
- 保存批注后，清理当前选区并将新批注加入右侧托盘

### 右侧批注托盘

- 空状态明确提示“先在代码区选中内容并添加标注”
- 每条批注卡片至少展示：
  - 行号范围
  - 代码摘录
  - 批注意见
  - 编辑 / 删除操作
- 新增成功后，对应卡片做一次短暂强调
- 编辑某条批注时，对应卡片进入 active 状态
- 清空当前文件批注时退出编辑态与聚焦态

### 发送区

- 保留可选问题输入框
- 发送按钮文案更明确表达“发送当前文件批注到 AI CLI”
- 会话未启动时继续禁用发送按钮，但提示文案更直接
- 发送成功后延续当前行为：清掉当前文件已发送批注，并给出轻提示

## 视觉规则

### 代码头部

- 更像“文件上下文条”，而不是普通标题条
- 保留当前 token 系统，不另起视觉主题
- 文件名优先，元信息次级显示

### 代码行

- 使用轻度行节奏与 gutter 对比增强可读性
- 不做重底色块，不做复杂 hover 系统
- 当前选区关联区域可使用极轻的背景或边缘提示

### 浮动按钮

- 采用轻按钮或轻芯片形态
- 与当前 surface / border / radius token 对齐
- 位置更贴近选区，但不覆盖阅读主路径

### 批注卡片

- 采用轻卡片结构，强化内部信息顺序
- 活动态和最近加入态只做一次轻提示，不制造强视觉闪烁
- 摘录区继续使用等宽字体，但增强层次与边界

## 错误处理

- 无文件时，右侧面板不显示发送型空状态，而是提示先选文件
- 删除正在编辑的批注时，同时退出编辑态
- 切换文件时清除 `activeAnnotationId` 与 `editingAnnotationId`
- 发送失败时沿用现有 toast 提示，不新增右侧错误展示区
- 会话未启动时不自动改造启动逻辑，仅优化禁用提示的可理解性

## 测试

### 单元 / 组件测试

- 新增批注后，当前文件批注列表出现对应卡片
- 新增批注后，`recentlyAddedAnnotationId` 会驱动最近加入态
- 点击编辑某条批注后，`activeAnnotationId` 正确切换
- 删除正在编辑的批注时，编辑态和活动态同步清理
- 无文件 / 无批注 / 会话未启动三类状态下，面板文案和按钮禁用正确

### 手动验证

- 打开 repo-view，选择文件，确认代码头部和代码行层次更清晰
- 选中文本，确认浮动按钮更低干扰但依旧可发现
- 添加批注，确认右侧卡片出现且有短暂强调
- 点击右侧编辑，确认对应卡片聚焦且代码区仍有上下文感
- 会话未启动时确认发送按钮禁用和提示清晰
- 发送成功后确认当前文件批注被清空，其余文件未发送批注不受影响

## 风险与取舍

- 不引入语法高亮意味着代码阅读质感提升主要依赖排版、间距和层次，收益有限但成本最低
- 增加 `activeAnnotationId` / `recentlyAddedAnnotationId` 会让窗口层状态略复杂，但仍处于可控范围
- 若视觉强化过度，容易偏离“克制轻量”目标，因此应优先保证阅读连续性

## 决策记录

| 决策 | 备选 | 选定 | 理由 |
|------|------|------|------|
| 视觉方向 | 仅美化代码区 / 代码区+右栏 / 三栏联动 | 代码区 + 右栏 | 用户明确要求聚焦批注链路 |
| 审阅节奏 | 快速批注 / 深度审阅 / 折中 | 快速批注 | 优先不打断阅读 |
| 产品风格 | 明显审阅模式 / 保持现有风格 | 保持现有风格 | 避免局部设计语言割裂 |
| 方案选择 | A 更克制 / B 更强调链路 | A | 更符合“轻量、安静、低打扰”目标 |
