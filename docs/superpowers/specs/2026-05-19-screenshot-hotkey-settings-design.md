# 全局截图快捷键设置设计

## 背景

当前截图能力通过 `electron.globalShortcut` 注册固定快捷键 `CommandOrControl+Shift+A`。该组合键在部分用户环境里会与其他应用冲突，而且没有可配置入口。与此同时，现有“AI 设置”弹窗实际已经承担了运行期配置入口的角色，继续单独新增页面会增加心智负担。

本设计将“AI 设置”统一重命名为“设置”，并在同一弹窗内新增“全局截图快捷键”配置区。截图快捷键属于整个应用级能力，不应跟项目级 AI CLI 配置混存。

## 目标

- 用户可以在“设置”里修改全局截图快捷键
- 用户可以禁用全局截图快捷键
- 用户可以一键恢复默认快捷键 `CommandOrControl+Shift+A`
- 保存后立即生效，无需重启应用
- 注册失败时保留旧的可用快捷键，不把配置写坏

## 非目标

- 不实现快捷键录制器，首版使用文本输入
- 不实现系统级冲突扫描，只以 Electron 实际注册结果为准
- 不把截图快捷键做成项目级配置
- 不新增单独的“截图设置”窗口

## 现状

### 相关代码

- `electron/screenshot/manager.ts`
  - 负责注册/注销全局截图快捷键
  - 当前内部写死 `CommandOrControl+Shift+A`
- `electron/main.ts`
  - 在应用生命周期内调用截图快捷键注册/注销逻辑
- `src/components/AiSettingsDialog.tsx`
  - 当前弹窗仅管理项目级 AI CLI 配置
- `src/App.tsx`
  - 顶部入口与命令入口当前文案为“AI 设置”

### 现有问题

1. 快捷键固定，无法避开其他软件冲突
2. 快捷键是全局能力，但当前没有应用级设置模型
3. “AI 设置”命名过窄，已经不适合继续承载通用设置项

## 方案概览

采用“单弹窗、双数据域”方案：

- 项目级设置：继续保存 AI CLI 配置，行为不变
- 应用级设置：新增全局截图快捷键配置，单独持久化

保存时由渲染层一次提交两类配置，但底层通过不同 IPC 通道分别落盘。截图快捷键变更由主进程即时重新注册 `globalShortcut`，失败则自动回滚。

## 数据模型

新增应用级设置文件，例如放在现有应用数据根目录下，与其他全局设置并列：

```json
{
  "screenshotShortcutEnabled": true,
  "screenshotShortcut": "CommandOrControl+Shift+A"
}
```

字段定义：

- `screenshotShortcutEnabled: boolean`
  - `true` 表示尝试注册全局截图快捷键
  - `false` 表示禁用，不执行注册
- `screenshotShortcut: string`
  - 保存用户最近一次设置的快捷键字符串
  - 即使 `enabled=false` 也保留，用于后续重新启用时恢复

默认值：

- `screenshotShortcutEnabled = true`
- `screenshotShortcut = CommandOrControl+Shift+A`

## 主进程设计

### 设置存储

新增一份应用级 settings 模块，职责仅包含：

- 读取全局设置
- 写入全局设置
- 缺省值合并

该模块不依赖项目 ID，不写入 `project.json`。

### 快捷键注册流程

截图快捷键的主进程应用逻辑统一为一个入口，例如：

1. 读取当前生效配置
2. 注销旧快捷键
3. 若 `enabled=false`，则直接结束
4. 若 `enabled=true`，尝试注册新快捷键
5. 成功则持久化新配置
6. 失败则恢复旧快捷键，并向调用方返回错误

### 为什么要先注册再写入

如果先写入配置、后注册快捷键，一旦注册失败，磁盘上的值就会变成“不可用配置”。下次应用启动时仍会继续失败。先验证注册、成功后再落盘，可以保证磁盘中永远保存最后一个已知可用状态。

### 应用启动行为

应用启动时：

1. 读取全局截图快捷键设置
2. 如果启用，则注册保存的快捷键
3. 如果禁用，则不注册任何截图快捷键
4. 如果磁盘值异常或缺失，则回退到默认值

## IPC 设计

新增应用级设置 IPC，避免复用项目设置通道：

- `settings:get-app-settings`
- `settings:set-app-settings`

返回值需要显式区分成功和错误，例如：

```ts
{ ok: true, value: AppSettings }
{ ok: false, error: string }
```

`set` 接口在截图快捷键场景下需要具备副作用：

- 不是单纯写配置
- 而是“尝试应用并保存”

也就是说，渲染层调用保存时，主进程会同步完成注册、回滚和落盘。

## 渲染层与界面设计

### 命名调整

所有用户可见的“AI 设置”统一改名为“设置”，包括：

- 顶部工具栏按钮
- 命令面板入口
- 弹窗标题

组件内部文件名可暂不改动，避免引入额外重命名噪音；首版优先改用户可见文案。

### 弹窗结构

“设置”弹窗保留单个保存按钮，内部拆成两个区块：

1. `AI CLI`
   - 保留现有主会话 AI CLI 配置
   - Repo View 相关隐藏逻辑保持原状
2. `截图快捷键（全局）`
   - 新增启用开关
   - 新增快捷键输入框
   - 新增“恢复默认”按钮
   - 新增一行帮助说明

### 交互细节

- 默认显示：
  - 开关为开启
  - 输入框为 `CommandOrControl+Shift+A`
- 关闭开关：
  - 输入框置灰
  - 但保留当前值，不清空
- 修改输入框：
  - 允许用户直接输入 Electron accelerator 字符串
- 点击“恢复默认”：
  - 将值恢复为 `CommandOrControl+Shift+A`
  - 自动把开关切回开启
- 点击“保存”：
  - AI CLI 配置和全局截图设置一起提交
  - 任一失败，弹窗保持打开并显示错误

### 帮助文案

建议使用简洁说明：

- `默认：Ctrl/Cmd + Shift + A`
- `示例：CommandOrControl+Shift+A`
- `保存后立即生效，对所有项目生效`

## 校验与错误处理

### 渲染层校验

渲染层只做最低限度校验：

- 启用状态下不能为空
- 去掉首尾空格

不在前端自行判断是否合法，因为真正的有效性由 Electron 运行时决定。

### 主进程校验

主进程以“尝试注册”为唯一真值来源：

- 字符串非法
- 快捷键已被占用
- 系统不允许注册

以上情况统一视为注册失败。

### 失败回滚

注册失败时：

1. 恢复旧快捷键注册状态
2. 不写入新的磁盘配置
3. 返回简洁错误给界面

示例错误文案：

- `快捷键不能为空`
- `快捷键注册失败，请更换未被占用的组合键后重试`
- `已恢复为之前的可用快捷键`

## 测试

### 自动化测试

- 应用级设置读取缺省值
- 启用默认快捷键时成功注册
- 禁用状态下不注册
- 修改快捷键时会注销旧值并注册新值
- 新值注册失败时恢复旧值
- “恢复默认”保存后恢复默认快捷键
- 入口与标题文案从“AI 设置”变为“设置”

### 手动测试

1. 打开“设置”，修改为未占用快捷键，保存后立即可触发截图
2. 修改为明显冲突或非法值，保存失败，原快捷键仍可用
3. 关闭全局截图快捷键，保存后快捷键不再生效
4. 点击“恢复默认”，保存后默认快捷键再次生效
5. 切换项目后再次打开“设置”，截图快捷键仍为同一全局值

## 决策记录

| 决策 | 选项 | 选定 | 理由 |
|------|------|------|------|
| 快捷键配置作用域 | 项目级 / 应用级 | 应用级 | `globalShortcut` 本身是应用级能力 |
| 配置入口 | 新页面 / 合并到现有弹窗 | 合并到现有弹窗 | 用户心智最小，改名为“设置”即可扩容 |
| 输入方式 | 文本输入 / 快捷键录制器 | 文本输入 | 首版复杂度低，足以解决冲突问题 |
| 生效方式 | 重启后生效 / 保存后即时生效 | 即时生效 | 用户反馈更直接，配置可立即验证 |
| 失败处理 | 保存配置但允许失效 / 回滚旧值 | 回滚旧值 | 保证配置文件只保存最后一个可用状态 |
