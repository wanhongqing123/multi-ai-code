# 项目构建设计

## 背景

当前应用已经有较完整的“项目 + target_repo + AICLI + Repo View + Diff 审查”工作流，但还缺少“项目构建”能力。实际使用中，不同项目的构建方式差异很大：

- 有的项目通过脚本构建
- 有的项目通过 `msbuild` / `.sln` 构建
- 有的项目通过 `cmake` / `gn` 等命令链构建
- 有的项目需要 `MSYS` 环境
- 同一个项目中，`SDK` 和 `Demo` 的构建方式也可能不同

这意味着“项目构建”不能做成固定模板枚举，也不能假设每个项目只有单一步骤。首版需要支持：

- 每个项目只有一条默认构建配置
- 这条默认配置里允许多个顺序步骤
- 每个步骤独立声明自己的执行环境和工作目录
- 构建失败后允许用户手动把失败上下文发送给 AICLI 分析，但 AICLI 只做原因分析，不改代码

## 目标

- 为每个项目提供一条默认构建流水线配置
- 支持多个顺序步骤，例如先构建 `SDK`，再构建 `Demo`
- 每个步骤可独立选择执行环境与工作目录
- 首版支持两种执行环境：
  - `MSYS`
  - `Visual Studio`
- 在主界面提供独立的构建面板，展示步骤状态与完整日志
- 任一步失败后立即停止后续步骤
- 失败后提供“分析失败原因”按钮，将失败上下文发送给 AICLI
- AICLI 分析时只诊断原因，不修改代码

## 非目标

- 不支持每个项目多套命名构建配置
- 不支持步骤并行执行
- 不支持“某一步失败也继续”
- 不支持自动识别 `cmake` / `gn` / `msbuild` 等项目类型
- 不支持自动安装、修复 `MSYS` 或 Visual Studio 环境
- 不支持 `Xcode` / `xcodebuild` 首版落地
- 不在首版为 `msbuild`、`cmake`、`gn` 单独做结构化专用字段

## 现状

### 项目级元数据

当前每个项目已有 `project.json`，用于保存项目级配置，例如：

- `target_repo`
- `msys_enabled`
- `ai_settings`
- `repo_view_ai_settings`

因此“项目构建”最自然的落点也是 `project.json`，而不是另建全局配置文件。

### 现有能力复用

当前工程已经具备如下基础能力，可直接复用到构建功能：

- 项目级 `project.json` 读写链路
- Windows 下 `MSYS` 检测与环境准备逻辑
- PTY 子进程与流式日志输出
- 主进程 / 渲染层 IPC 通道
- 主界面中的长任务展示区域与状态按钮模式

## 设计概览

首版采用“项目级默认构建流水线 + 独立构建面板 + 手动失败分析”方案：

1. 在 `project.json` 中新增一条默认构建配置 `build_config`
2. `build_config` 内部包含多个顺序步骤
3. 每个步骤独立配置：
   - 步骤名称
   - 环境类型
   - 工作目录
   - 命令
   - 启用状态
4. 主界面新增独立“构建”入口与构建面板
5. 构建由主进程顺序执行；任一步失败即停止
6. 失败后由用户手动点击“分析失败原因”触发 AICLI 诊断

## 一、数据模型

### 项目级默认构建配置

在每个项目的 `project.json` 中新增：

```json
{
  "build_config": {
    "enabled": true,
    "steps": [
      {
        "name": "Build SDK",
        "envType": "msys",
        "cwd": "sdk",
        "command": "./build_sdk.sh",
        "enabled": true
      },
      {
        "name": "Build Demo",
        "envType": "visual-studio",
        "cwd": "demo",
        "command": "msbuild demo.sln /p:Configuration=Release /p:Platform=x64",
        "enabled": true
      }
    ]
  }
}
```

### 字段定义

- `build_config.enabled: boolean`
  - 是否启用项目构建功能
- `build_config.steps: BuildStep[]`
  - 默认构建流水线中的顺序步骤列表

每个 `BuildStep` 包含：

- `name: string`
  - 步骤名称，用于 UI 和日志
- `envType: 'msys' | 'visual-studio'`
  - 步骤执行环境
- `cwd: string`
  - 相对 `target_repo` 的工作目录
- `command: string`
  - 该步骤实际执行的一整条命令
- `enabled: boolean`
  - 是否参与当前默认构建流水线

### 设计约束

- 每个项目只有一条默认构建配置，不做多套命名方案
- 步骤顺序严格按数组顺序执行
- `cwd` 统一解释为相对 `target_repo`
- 首版不额外增加 `solutionPath`、`configuration`、`platform` 等专用字段；都写进 `command`

## 二、执行模型

### 总体执行流程

点击“开始构建”后：

1. 读取当前项目的 `build_config`
2. 校验配置是否合法
3. 过滤掉 `enabled=false` 的步骤
4. 按顺序逐步执行
5. 任一步失败则整体构建进入 `failed`
6. 所有步骤成功则整体构建进入 `succeeded`

### 步骤执行环境

每个步骤按 `envType` 分别准备环境：

#### `msys`

适用于：

- `bash` 脚本
- `configure` / `make`
- 依赖 Unix 工具链的构建命令

执行原则：

- 主进程先检测本机 `MSYS` 是否可用
- 将步骤 `cwd` 解析成 `target_repo` 下绝对路径
- 在该目录下通过 `bash -lc "<command>"` 等方式执行
- 全量采集输出并写入构建日志

#### `visual-studio`

适用于：

- `msbuild xxx.sln`
- `devenv /Build`
- 依赖 `cl` / `link` / `nmake` 的构建链路

执行原则：

- 主进程先检测本机 Visual Studio / Build Tools 是否可用
- 先初始化开发者命令行环境
- 再切换到步骤 `cwd`
- 执行步骤 `command`
- 全量采集输出并写入构建日志

### 失败策略

- 任一步失败后立即停止
- 后续步骤全部不再执行
- 不支持首版中的“失败也继续”
- 用户主动点击“停止构建”时中断当前步骤并结束本次 run

## 三、构建面板

### 独立于 AICLI

构建面板与 AICLI 分离，不复用主会话终端区域。原因：

- 构建是顺序流水线任务，不是对话会话
- 需要明确展示每一步状态
- 后续要支持“重新构建 / 停止 / 失败分析 / 复制日志”等构建专属交互

### 面板内容

构建面板首版包含：

- 当前项目名称
- 默认构建配置摘要
- `开始构建` 按钮
- `停止构建` 按钮
- 当前整体状态
- 步骤列表与每一步状态
- 完整构建日志区

### 步骤状态

每个步骤状态收敛为：

- `pending`
- `running`
- `succeeded`
- `failed`
- `skipped`

整体构建状态收敛为：

- `idle`
- `running`
- `succeeded`
- `failed`
- `stopped`

## 四、项目设置中的构建配置表单

### 配置入口

在项目设置中新增“项目构建”区块，用于编辑该项目唯一的一条默认构建流水线。

### 表单结构

包含：

- `启用项目构建` 开关
- 步骤列表

每个步骤编辑项包含：

- `步骤名称`
- `环境类型`
- `工作目录`
- `命令`
- `启用/禁用`

列表操作：

- `新增步骤`
- `删除步骤`
- `上移`
- `下移`

### 无配置时的行为

如果项目未配置默认构建流水线，点击“构建”入口时提示：

- `当前项目未配置默认构建步骤，请先到项目设置中配置`

## 五、失败分析与 AICLI 对接

### 触发方式

构建失败后，在构建面板中显示：

- `分析失败原因`

该按钮为用户手动触发，不自动分析。

### 发送给 AICLI 的上下文

触发分析时，发送给 AICLI 的内容至少包括：

- 项目名
- `target_repo`
- 失败步骤名称
- 失败步骤环境类型
- 失败步骤工作目录
- 失败步骤命令
- 失败步骤退出码
- 失败步骤尾部日志
- 其余步骤的摘要状态

### 固定约束

发送给 AICLI 时附带固定约束：

- 只分析失败原因
- 不修改代码
- 不执行修复
- 不生成补丁
- 优先判断属于：
  - 环境缺失
  - 依赖缺失
  - 命令错误
  - 路径错误
  - 脚本错误
  - 编译错误

### 期望输出

期望 AICLI 返回面对用户可读的分析结果，至少包含：

- 失败归类
- 最可能原因
- 证据
- 建议先检查什么

## 六、失败边界

首版明确区分四类失败：

### 1. 配置错误

例如：

- 未配置 `build_config`
- 步骤列表为空
- 步骤命令为空
- `cwd` 不存在或非法

这类错误在构建启动前拦截，不进入真正执行态。

### 2. 环境错误

例如：

- `MSYS` 不存在
- Visual Studio / Build Tools 不存在
- 无法初始化对应 shell

这类错误在步骤执行前报出，并明确提示是“环境未就绪”。

### 3. 命令失败

环境已就绪，但命令返回非零退出码。

这类错误才视为标准构建失败，并允许点击“分析失败原因”。

### 4. 执行中断

例如：

- 用户手动停止
- PTY / 子进程异常退出

这类不自动归类为“构建失败原因分析”对象。

## 七、模块与职责

### 主进程

建议新增或扩展以下能力：

- 项目构建配置读写
- 构建执行器
- 构建日志流式广播
- 构建停止
- 构建失败上下文缓存

### 渲染层

建议新增：

- 项目设置中的“项目构建”配置区块
- 主界面的构建入口
- 独立构建面板
- 失败后“分析失败原因”按钮

### 数据持久化

项目构建配置保存在项目 `project.json` 中，不新建额外项目配置文件。

## 八、测试方案

### 配置模型测试

1. `build_config` 可正确读写到 `project.json`
2. 多步骤顺序保持不变
3. `enabled=false` 的步骤不会执行

### 主进程执行测试

1. `msys` 步骤使用正确的 `cwd` 和命令执行
2. `visual-studio` 步骤使用正确的 `cwd` 和命令执行
3. 任一步失败后后续步骤不再执行
4. 停止构建能中断当前步骤

### UI 状态测试

1. 没有构建配置时能给出明确提示
2. 构建中按钮状态正确
3. 步骤状态正确流转
4. 失败后显示“分析失败原因”

### AICLI 对接测试

1. 只有 `failed` 状态时按钮可点击
2. 发给 AICLI 的上下文包含失败步骤、退出码和尾部日志
3. 约束文案明确为“只分析，不改代码”

## 九、实施顺序建议

建议按以下顺序实现：

1. 项目级 `build_config` 数据模型与保存
2. 主进程构建执行器
3. 构建面板 UI
4. “分析失败原因”与 AICLI 对接
5. 手动验证 `MSYS` / `Visual Studio` 两类真实项目

## 决策记录

| 决策 | 选项 | 选定 | 理由 |
|------|------|------|------|
| 配置粒度 | 多套命名构建配置 / 每项目一条默认配置 | 每项目一条默认配置 | 首版需求明确要求每项目只需一条 |
| 流水线结构 | 单步骤 / 多顺序步骤 | 多顺序步骤 | 同一项目可能先构建 SDK 再构建 Demo |
| 步骤环境 | 整条配置共用环境 / 每步独立环境 | 每步独立环境 | 允许 `MSYS -> Visual Studio` 混合链路 |
| 首版环境支持 | 通用全部平台 / 先支持 Windows 两类环境 | `MSYS` + `Visual Studio` | 先覆盖当前最强需求 |
| 构建输出位置 | 复用 AICLI / 独立构建面板 | 独立构建面板 | 构建与对话语义不同，便于后续扩展 |
| 失败分析触发 | 自动触发 / 手动按钮触发 | 手动按钮触发 | 用户可先看日志，避免无意义分析 |
| AICLI 分析行为 | 可顺手修代码 / 仅分析原因 | 仅分析原因 | 用户明确要求此时不改代码 |
