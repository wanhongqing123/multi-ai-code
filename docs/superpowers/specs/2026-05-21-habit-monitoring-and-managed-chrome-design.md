# 习惯监控与托管 Chrome 设计

## 背景

当前应用已经具备较完整的项目工作流能力，包括：

- 项目 / `target_repo`
- 方案与主会话 AICLI
- Repo View
- Diff 审查
- 构建与体检

同时，当前产品里还保留着三块与“效率沉淀”相关但定位分散的能力：

- 顶栏 `模板`
- 顶栏 `向导`
- `Skill 学习`

现状存在三个明显问题：

1. `模板` 与 `向导` 使用频率低，继续放在顶栏会稀释主工作流。
2. `Skill 学习` 当前更偏向“基于事件生成 prompt 模板候选”，价值过窄，无法覆盖“自动识别常用网站流程、自动整理常用操作路径、自动个性化界面”这类更直接的用户价值。
3. 现有产品无法采集网站使用习惯，也没有“由本应用启动并托管的 Chrome 会话”这一产品能力。

用户的核心诉求不是继续“学习模板”，而是让应用持续监控自己的使用习惯，并自动沉淀为更直接的智能化流程，例如：

- 经常访问的网站与固定页面
- 经常执行的站点内导航流程
- 在应用内经常反复执行的操作路径
- 可自动前置、隐藏或弱化的界面入口

因此，本次设计的目标是将 `Skill 学习` 重构为 `习惯监控`，并新增“托管 Chrome + 双源习惯采集 + 低风险流程默认自动启用”的完整能力闭环。

## 目标

- 将 `Skill 学习` 升级为统一的 `习惯监控` 中心。
- 同时采集两类习惯来源：
  - 本应用内操作行为
  - 由本应用启动的托管 Chrome 会话中的网站访问与操作行为
- 提供托管 Chrome 的明确产品入口：
  - 顶栏快速入口
  - `习惯监控` 面板内的会话控制入口
- 将重复习惯自动沉淀成三类结果：
  - 常用流程
  - 界面个性化调整
  - 高风险候选
- 对低风险流程与轻量界面调整默认自动生效。
- 提供配置项，允许用户关闭自动采集、自动启用和自动个性化能力。
- 弱化或移除 `模板`、`向导` 的顶层入口。

## 非目标

- 不采集用户平时自行打开、且并非由本应用启动的 Chrome 实例。
- 不在第一版支持 Firefox、Edge、Safari 或 `Xcode` 相关浏览器托管能力。
- 不在第一版记录敏感输入原文，例如密码、Cookie、Token、支付信息、任意表单正文。
- 不在第一版自动启用高风险站点操作，例如提交、支付、删除、发布、登录类动作。
- 不在第一版做深层产品逻辑自修改，只允许轻量界面个性化调整。
- 不要求保留 `模板`、`向导` 的顶栏可见性；它们可以退居二级入口或后续彻底移除。

## 现状

### 现有 UI 入口

当前主界面顶栏仍然存在以下入口：

- `模板`
- `Skill 学习`
- `向导`

相关代码入口位于：

- `src/App.tsx`
- `src/components/TemplatesDialog.tsx`
- `src/habit/SkillStudioDialog.tsx`

### 现有习惯学习实现

当前“Skill 学习”的实现主要由以下模块组成：

- `src/habit/SkillStudioDialog.tsx`
- `src/habit/CollectionSettingsPanel.tsx`
- `src/habit/FirstRunNoticeDialog.tsx`
- `electron/habit/generator.ts`
- `electron/habit/db.ts`
- `electron/store/db.ts`

现有数据层主要围绕两张表：

- `habit_events`
- `skill_candidates`

现有生成链路更偏向：

- 记录模板调用和若干产品事件
- 聚类重复行为
- 生成 prompt 模板候选

这套链路不足以表达“站点流程”、“界面个性化调整”、“低风险自动流程”。

### 托管浏览器现状

当前产品里没有“由本应用启动并托管外部 Chrome”的产品功能。

这意味着：

- 不能采集真实网站访问习惯
- 不能基于网站访问与站内导航生成常用流程
- 不能在产品层显式展示浏览器会话状态

## 设计概览

本次采用“`习惯监控` 中心 + 托管 Chrome + 双源习惯采集 + 风险分层自动启用”的方案：

1. 将 `Skill 学习` 重命名并重构为 `习惯监控`。
2. 从顶栏移除或显著弱化 `模板`、`向导`。
3. 在顶栏新增 `托管 Chrome` 快速入口。
4. 本应用启动一个受管 Chrome 实例，并通过 Chrome DevTools Protocol 连接。
5. 同时采集：
   - 应用内行为
   - 托管 Chrome 行为
6. 将重复行为聚类为：
   - 常用流程
   - 界面个性化调整
   - 高风险候选
7. 低风险结果默认自动启用，并在 `习惯监控` 中提供关闭入口。

## 一、产品结构调整

### 顶栏结构

顶栏建议调整为：

- 保留：项目、方案、设置、构建、Diff 审查、体检等主工作流入口
- 新增：`托管 Chrome`
- 将 `Skill 学习` 替换为 `习惯监控`
- 移除或弱化：`模板`、`向导`

### `模板` 与 `向导` 的处理策略

第一版建议：

- `模板`
  - 从顶栏移除
  - 现有数据暂不删除
  - 若仍需保留，退至二级入口，例如设置或低频功能区
- `向导`
  - 从顶栏移除
  - 可退至帮助、设置或首次启动场景

产品层应体现出：

- `模板` 与 `向导` 不再是主工作流入口
- 用户主要通过 `习惯监控` 获取沉淀能力

### `习惯监控` 面板结构

`习惯监控` 面板建议分为四个区块：

1. `采集`
   - 采集应用内习惯
   - 采集托管 Chrome 习惯
   - 数据清理与保留策略
2. `托管 Chrome`
   - 当前状态
   - 启动 / 停止
   - 当前活动站点
   - 最近采集状态
3. `自动生效`
   - 自动启用低风险流程
   - 自动个性化界面
4. `已生效项目`
   - 已启用流程
   - 已应用的界面调整
   - 每项都可关闭

## 二、托管 Chrome

### 启动方式

第一版采用“由本应用启动并托管专用 Chrome 实例”的方案，而不是接管用户平时手动打开的系统 Chrome。

启动原则：

- 本应用寻找本机 Chrome 可执行文件
- 启动一个专用 Chrome 进程
- 启动参数至少包括：
  - `--remote-debugging-port=<port>`
  - `--user-data-dir=<app-managed-profile>`

这样可以保证：

- 会话生命周期由本应用管理
- 浏览器 profile 与用户日常 Chrome 隔离
- 本应用可以通过 Chrome DevTools Protocol 稳定采集导航与页面行为

### 产品入口

托管 Chrome 需要两个入口：

1. 顶栏 `托管 Chrome`
   - 未启动时：启动并打开托管 Chrome
   - 已启动时：聚焦现有托管 Chrome 会话
2. `习惯监控` 面板中的会话卡片
   - 启动
   - 停止
   - 显示端口、活动站点、采集状态
   - 清理独立 profile
   - 暂停网站习惯采集

### 采集范围

托管 Chrome 第一版允许采集：

- 访问了哪些网站
- URL path 级别的访问轨迹
- 标签页切换
- 页面内主要点击与导航流程
- 非敏感输入行为的“类型提示”，例如“在搜索框输入关键词”
- 停留时间与回访频率

第一版不采集或不落盘明文：

- 密码
- Cookie
- Token
- 支付信息
- 敏感表单正文
- 任意输入框中的完整原始文本

### 事件模型

托管 Chrome 事件建议至少包含：

- `session_id`
- `tab_id`
- `url`
- `origin`
- `action_type`
  - `navigate`
  - `click`
  - `input`
  - `submit`
  - `switch-tab`
- `element_hint`
  - 文本摘要 / `aria-label` / `role` / 选择器摘要
- `timestamp`

第一版不建议把完整 CSS selector 作为唯一锚点，以免页面稍有改版就完全失效。

## 三、习惯来源与结果模型

### 双源习惯采集

习惯监控统一采集两类来源：

1. `app_ui`
   - 面板打开
   - 功能点击
   - 工作流切换
   - 构建、Diff、Repo View 等高频操作
2. `managed_chrome`
   - 站点访问
   - 标签页切换
   - 页面点击
   - 简单输入提示
   - 导航链路

### 输出结果分层

习惯监控的输出不再只是一类“模板候选”，而是三类结果：

1. `常用流程`
   - 可直接执行的轻自动化
   - 例如：启动托管 Chrome 并打开固定站点页面
   - 例如：打开项目后自动显示构建面板
2. `界面个性化调整`
   - 不执行任务，只调整产品呈现
   - 例如：隐藏 `模板`
   - 例如：隐藏 `向导`
   - 例如：将 `托管 Chrome`、`构建` 前置
3. `高风险候选`
   - 可以识别并展示
   - 默认不自动启用

### 常用流程与旧模板能力的关系

第一版建议：

- `常用流程` 成为主输出
- 原有 prompt 模板能力降为次级结果
- 只有少数确实适合抽象成 prompt 的重复行为，才继续沉淀为模板型 skill

也就是说，主概念从“学 skill / 学模板”变成“监控习惯并自动沉淀流程”。

## 四、风险分层与自动启用

### 低风险

可默认自动启用的低风险动作包括：

- 打开某个本地面板
- 打开托管 Chrome
- 导航到某个站点或固定路径
- 恢复布局与上下文
- 隐藏低频入口
- 调整按钮顺序
- 打开项目后自动展示高频面板

### 高风险

不得默认自动启用的动作包括：

- 页面提交
- 发布
- 删除
- 登录 / 登出 / 切账号
- 发送消息
- 下单 / 支付
- 任何可能修改远端状态的站点交互
- 任何可能改动本地仓库或代码的自动动作

### 启用策略

第一版启用策略：

- 低风险流程：默认自动生效
- 低风险界面调整：默认自动生效
- 高风险候选：只展示，不自动执行

风险规则应由产品逻辑显式判定，不把第一版风险判断完全交给模型自由决策。

## 五、配置与可见性

### 默认值

第一版默认开启：

- 采集应用内习惯
- 采集托管 Chrome 习惯
- 自动启用低风险流程
- 自动个性化界面

### 用户可关闭项

虽然默认生效，但配置界面必须提供关闭能力。建议至少提供：

- `采集应用内习惯`
- `采集托管 Chrome 习惯`
- `自动启用低风险流程`
- `自动个性化界面`

### 已生效项目列表

`习惯监控` 中必须清楚显示：

- 哪些流程已经自动启用
- 哪些界面调整已经生效
- 每一项都可以关闭

产品策略为：

- 默认生效
- 用户无需逐条确认
- 但任何已生效项都应可回退或关闭

## 六、数据模型

### 原始事件层

保留并扩展现有 `habit_events`，新增更细事件来源与类型。

建议扩展字段语义：

- `source`
  - `app_ui`
  - `managed_chrome`
- `kind`
  - `panel_open`
  - `action_triggered`
  - `site_visit`
  - `site_click`
  - `site_input_hint`
  - `tab_switch`
  - 兼容已有 `template_used` 等旧类型

### 习惯流程层

新增 `habit_flows`，用于承载从原始事件聚类出的结果。

建议字段：

- `id`
- `kind`
  - `app-flow`
  - `site-flow`
  - `ui-adjustment`
- `title`
- `summary`
- `evidence_count`
- `risk_level`
  - `low`
  - `high`
- `enabled_by_default`
- `status`
  - `candidate`
  - `active`
  - `disabled`
- `payload`
  - 具体流程或界面调整的结构化动作

### 托管 Chrome 会话层

新增 `managed_chrome_sessions`，记录受管浏览器会话状态。

建议字段：

- `id`
- `port`
- `profile_dir`
- `started_at`
- `last_active_at`
- `running`
- `last_active_url`

## 七、模块与职责

### 渲染层

建议新增或改造：

- `SkillStudioDialog` 重构为 `HabitMonitorDialog`
- 顶栏 `托管 Chrome` 按钮
- `习惯监控` 面板中的：
  - 采集开关
  - 会话控制
  - 已启用流程
  - 已应用界面调整

### 主进程

建议新增：

- 托管 Chrome 会话管理器
  - 查找 Chrome
  - 启动 Chrome
  - 停止 Chrome
  - 管理 profile
- Chrome CDP 采集桥接
- 习惯聚类与结果生成器
- 自动启用与自动个性化决策器

### 数据层

建议在现有 `habit_events` / `skill_candidates` 基础上扩展或演化：

- `habit_events`
- `habit_flows`
- `managed_chrome_sessions`

是否保留 `skill_candidates`：

- 可以第一版兼容保留
- 但产品语义上不应再作为中心数据模型

## 八、失败边界与安全约束

### 托管 Chrome 边界

第一版只允许监控：

- 本应用启动的受管 Chrome

第一版不允许监控：

- 用户手动启动、且未被本应用管理的普通 Chrome 会话

### 隐私边界

不得记录或持久化：

- 密码
- Token
- Cookie
- 支付信息
- 敏感输入原文

### 自动化边界

不得默认自动执行：

- 提交
- 发布
- 删除
- 登录 / 登出
- 消息发送
- 支付
- 任何改动远端状态的站点行为

### 产品修改边界

第一版允许的“自动调整和修复”仅限：

- 隐藏或弱化低频入口
- 前置高频入口
- 自动启用低风险常用流程

第一版不允许：

- 自动重写更深层业务逻辑
- 自动变更 AI CLI 配置
- 自动触发代码改写

## 九、测试与验证

### 应用内习惯链路

1. 能识别高频应用内入口与操作路径
2. `模板`、`向导` 可被自动弱化或隐藏
3. 高频面板可被自动前置或自动打开

### 托管 Chrome 链路

1. 能从本应用启动与停止托管 Chrome
2. 能记录站点访问与简单导航路径
3. 能将重复站点流程沉淀为低风险常用流程

### 自动启用与关闭

1. 低风险流程默认生效
2. 界面个性化调整默认生效
3. 用户可在 `习惯监控` 中关闭任意已启用项

### 安全边界

1. 高风险站点操作不会被默认自动启用
2. 不记录敏感输入原文
3. 不采集非托管 Chrome 会话

## 决策记录

| 决策 | 选项 | 选定 | 理由 |
|------|------|------|------|
| 浏览器采集方式 | 采集用户普通 Chrome / 仅采集本应用托管 Chrome | 仅采集本应用托管 Chrome | 技术边界清晰、权限更可控、产品行为可解释 |
| 托管 Chrome 入口 | 仅顶部按钮 / 仅面板内入口 / 两者都有 | 两者都有 | 既满足快速启动，也满足状态可见性 |
| 自动启用策略 | 全量手动确认 / 低风险默认启用 | 低风险默认启用 | 用户明确要求低风险流程默认生效 |
| 自动调整范围 | 深层逻辑调整 / 轻量个性化调整 | 轻量个性化调整 | 风险更低，更符合第一版落地范围 |
| `模板` / `向导` 策略 | 保留顶栏 / 弱化或移除 | 弱化或移除 | 用户明确表示两者使用少，应退出主入口 |
| 主产品概念 | 保持“Skill 学习” / 升级为“习惯监控” | 升级为“习惯监控” | 用户真正需求是持续习惯采集与流程沉淀，而不是模板学习 |
