# 项目构建顺序执行与单步执行设计

## 背景

当前项目构建功能已经支持：

- 按项目保存多步骤构建配置
- 按配置顺序执行已启用步骤
- 在构建面板中展示步骤状态、实时日志与失败信息

但当前执行模型只有一种模式：整条流水线从上到下顺序执行。用户无法在构建面板中只运行某一个步骤，例如只单独执行 `生成 Demo`、`打包` 或 `拷贝产物`。

这会带来两个直接问题：

- 调试某个后置步骤时，必须重复运行前面所有步骤，耗时高
- UI 上“开始构建”是总控行为，但没有“单独执行当前步骤”的入口

本次设计要在保留现有顺序构建能力的前提下，新增“每个步骤可单独执行”的运行模式。

## 目标

- 保留现有整条流水线的顺序构建能力
- 支持在构建面板中对单个已启用步骤发起单独构建
- 单独构建时不自动补跑前置步骤，也不执行后续步骤
- UI 明确区分“未执行”和“跳过”
- 复用现有构建日志、失败上下文、停止构建和失败分析能力

## 非目标

- 不支持“从某一步开始向后连续执行”
- 不支持“一次选择多个步骤组成子流水线”
- 不改动现有步骤配置结构中的环境类型、编码、Visual Studio 实例绑定逻辑
- 不改变现有失败分析 prompt 的总体格式，只补充必要上下文

## 现状

### 启动模型

当前主进程 `build:start` 只接收项目 id，由主进程读取该项目的完整构建配置后传给 `buildRunner.start()`。

`buildRunner.start()` 默认将所有已启用步骤初始化为 `pending`，并在 `runBuild()` 中按顺序执行。

### 步骤状态

当前 `BuildStepStatus` 只有：

- `pending`
- `running`
- `succeeded`
- `failed`
- `skipped`

其中 `skipped` 目前同时承载了“中途停止后未继续执行”这类含义，但并不能表达“本次本来就不在执行范围内”。

### UI 入口

当前构建面板只有：

- `开始构建`
- `停止构建`
- `分析失败原因`

步骤卡片只有状态展示，没有“单独构建”入口。

## 设计概览

本次采用“统一运行器 + 显式执行范围”的方案：

1. `build:start` 从“固定全量构建”升级为“带执行范围的启动请求”
2. `buildRunner` 在同一套运行时中同时支持：
   - `scope = all`
   - `scope = single-step`
3. 步骤状态新增 `not-run`，用于表达“本次没有纳入执行范围”
4. 构建面板顶部保留总控按钮 `顺序构建`
5. 每个步骤卡片新增 `单独构建`

这样可以最大限度复用已有：

- 实时日志流
- 停止构建
- 失败记录
- 失败分析入口

同时避免拆出第二套构建运行器。

## 数据模型

### 启动请求

新增构建运行范围：

- `scope: 'all' | 'single-step'`
- `stepId?: string`

约束：

- `scope === 'all'` 时，`stepId` 必须为空
- `scope === 'single-step'` 时，`stepId` 必须存在，且必须命中当前项目配置中的某个步骤

### 运行时状态

`BuildRuntimeState` 新增：

- `scope: 'all' | 'single-step' | null`
- `requestedStepId: string | null`

用途：

- UI 可以区分最近一次运行是顺序构建还是单步构建
- 构建完成或失败后，仍能知道本次执行范围

### 步骤状态

`BuildStepStatus` 新增：

- `not-run`

状态语义统一如下：

- `not-run`：本次构建不在执行范围内
- `pending`：本次构建计划执行，但尚未开始
- `running`：正在执行
- `succeeded`：执行成功
- `failed`：执行失败
- `skipped`：本次原本计划执行，但因为停止构建或后续逻辑未继续执行

## 主进程接口设计

### build:start

主进程 `build:start` 从“仅接收项目 id”升级为接收：

- `projectId`
- `scope`
- `stepId`

主进程职责：

1. 读取项目元数据和构建配置
2. 校验 `target_repo` 存在且为目录
3. 校验执行范围是否合法
4. 组装完整 `StartBuildRequest`
5. 调用 `buildRunner.start(request)`

### 范围校验

校验规则：

- `all`：至少存在一个已启用步骤
- `single-step`：
  - `stepId` 必填
  - 步骤必须存在
  - 步骤必须 `enabled === true`

失败时直接返回结构化错误，不启动运行器。

建议错误信息：

- `build step is required for single-step scope`
- `build step not found: <stepId>`
- `build step is disabled: <stepId>`

## 运行器设计

### 初始化

`buildRunner.start()` 接收执行范围后，初始化步骤状态：

#### 顺序构建

- 已启用步骤：`pending`
- 未启用步骤：`skipped`

#### 单步构建

- 目标步骤：`pending`
- 其它所有步骤：`not-run`

这里的 `not-run` 包括：

- 已启用但未被本次选中的步骤
- 未启用步骤

原因是这两类步骤在“单步构建”语义下都不属于本次执行范围，UI 不应误导为“被跳过”。

### 执行循环

#### 顺序构建

保持现有行为：

- 从上到下执行所有 `enabled` 步骤
- 失败即终止
- stop 时终止当前步骤并把后续待运行步骤标记为 `skipped`

#### 单步构建

仅执行目标步骤：

1. 定位 `stepId`
2. 仅为该步骤调用 `spawnForStep()`
3. 成功则整次构建成功
4. 失败则整次构建失败

不允许：

- 自动补跑前置步骤
- 自动继续跑后续步骤

### 停止构建

单步构建时：

- 当前步骤被 stop 后，整体状态仍为 `stopped`
- 其它 `not-run` 步骤保持 `not-run`
- 不额外产生 `skipped`

顺序构建时：

- 当前步骤若中断，则后续原本待执行步骤标记为 `skipped`

## 日志与失败上下文

### 日志

日志模型继续复用当前单份日志缓冲区，不拆分。

新增文案层区分：

- `scope = all`：顺序构建
- `scope = single-step`：单步构建

日志本身仍按当前流式方式附加。

### 失败上下文

`BuildFailureContext` 保持现有主体结构不变，但建议补充：

- `scope: 'all' | 'single-step'`

用途：

- 失败分析时区分“整条流水线中的失败”与“单步构建中的失败”

如果当前实现不方便立即扩字段，也允许第一版先只依赖 `BuildRuntimeState.scope`，不阻塞主功能。

## 前端交互设计

### 顶部主按钮

构建面板顶部：

- `开始构建` 改为 `顺序构建`
- `停止构建` 保持不变
- `分析失败原因` 保持不变

### 步骤卡片

每个步骤卡片新增：

- `单独构建`

按钮行为：

- 点击后发起 `scope = single-step`
- 目标 `stepId = 当前步骤 id`

禁用规则：

- 当前已有构建运行中时，所有“顺序构建 / 单独构建”按钮禁用
- 当前步骤 `enabled === false` 时，“单独构建”禁用

### 步骤状态标签

新增 `not-run` 对应展示文案：

- `未执行`

其余状态文案保持现有风格：

- `等待中`
- `进行中`
- `成功`
- `失败`
- `跳过`

### 最近执行提示

若最近一次为单步构建，可在状态卡片中展示：

- `最近执行：<stepName>（单独构建）`

若最近一次为顺序构建，则保持现有总体状态展示即可。

## UI 与状态一致性要求

- 单步构建成功后，只有目标步骤显示 `成功`
- 单步构建失败后，只有目标步骤显示 `失败`
- 其它步骤统一显示 `未执行`
- 不允许把“单步构建没跑到的步骤”显示为 `跳过`
- 不允许把“未启用步骤”在单步构建视图中混成 `失败` 或 `等待中`

## 错误处理

### 启动前错误

主进程返回结构化错误，不进入运行态：

- 未选择项目
- 构建配置未就绪
- 构建配置未启用
- 没有已启用步骤
- 单步构建未传 `stepId`
- 单步构建目标步骤不存在
- 单步构建目标步骤未启用

### 运行期错误

运行期错误继续沿用现有行为：

- 当前步骤失败
- 整体状态切为 `failed`
- 记录 `lastFailure`

## 测试策略

### runner.test.ts

新增或调整测试：

- 顺序构建仍按已启用步骤从上到下执行
- 单步构建只执行目标步骤
- 单步构建时非目标步骤标记为 `not-run`
- 单步构建成功后整体状态为 `succeeded`
- 单步构建失败后整体状态为 `failed`
- 单步构建 stop 后整体状态为 `stopped`
- 顺序构建 stop 后未运行的后续步骤仍为 `skipped`

### ProjectBuildPanel.test.tsx

新增或调整测试：

- 顶部按钮显示 `顺序构建`
- 每个步骤卡片显示 `单独构建`
- `not-run` 状态显示为 `未执行`
- 运行中时所有启动类按钮禁用
- 禁用步骤的“单独构建”按钮不可点击

### App / 主进程接线测试

新增或调整测试：

- 点击顶部按钮时传 `scope = all`
- 点击步骤按钮时传 `scope = single-step` 和对应 `stepId`
- 主进程对无效 `stepId` 返回错误

## 兼容性与迁移

- 这是运行时协议扩展，不涉及现有项目构建配置文件结构变更
- 老项目无需迁移即可继续使用顺序构建
- 前端和主进程必须同时更新；旧前端无法驱动新接口中的单步构建能力

## 结论

本次功能通过为现有构建运行器增加“执行范围”而不是拆出第二套实现，能以最小的维护成本支持：

- 整条流水线顺序构建
- 任一步骤单独构建

同时通过新增 `not-run` 状态，明确区分“本次未执行”和“中途跳过”，保证构建面板的语义准确。
